# 병렬 리뷰 → 교차검증 → TDD 수정 + 적대적 검증 워크플로

Claude Code 워크플로/서브에이전트로 기존 코드베이스를 전면 검토·수정할 때 검증된 4단계 구성. ht_trading 전면 개선 세션(2026-07-04, 커밋 20여 개 → 라이브 배포)과 ht_dde 8-에이전트 서브시스템 분석(2026-07-04)에서 두 차례 실전 사용.

## 4단계 구성

1. **관점 분할 병렬 리뷰** — 검토를 관점별 병렬 에이전트로 분할 (예: ① 라이브 주문 경로 ② 전략·리스크 로직 ③ 구조·운영). 한 에이전트가 전부 보는 것보다 각자의 컨텍스트가 깊어진다.
2. **'높음' 판정만 교차검증** — 리뷰 결과를 종합하기 전에, 심각도 '높음' 항목만 **코드에서 직접 재확인** (리뷰 에이전트의 오탐 제거). 여기서 걸러진 것만 수정 대상.
3. **독립 파일셋 병렬 TDD 수정** — 결함들을 파일셋이 겹치지 않게 병렬 배분, 각 수정은 실패 테스트 먼저 작성 → 구현 → 통과. 논리 단위로 커밋 분리.
4. **적대적 검증 (리뷰 에이전트 분리)** — 수정 완료 후 별도 에이전트가 라이브 회귀 중심으로 검증: **콜사이트 전수 grep** (수정이 모든 호출 경로를 커버하는지) + **수정을 일시 되돌려 테스트가 실제로 실패하는지 실증** (테스트가 진짜 그 결함을 잡는지).

## 운영 함정 (실측)

- **파이프라인이 테스트 종료코드를 삼킨다** — `pytest ... > file 2>&1` 뒤 이어지는 명령이 성공하면 실패가 가려져, 5건 실패 상태로 커밋된 사고 발생. 에이전트가 "테스트 통과"를 커밋 전제로 삼을 때는 **`echo exit=$?` 로 종료코드를 별도 확인**해야 한다 (자가 발견 후 재발 방지 규칙을 lessons 에 기록).
- **리뷰 에이전트 무응답** — 병렬 에이전트 중 하나가 응답 없이 대기 상태가 될 수 있어 재요청 필요.
- **위험 명령의 사람 게이팅** — 실계좌 프로덕션 재시작(`launchctl unload/load`)은 권한 분류기가 자동 실행을 차단 → 코드 수정까지만 자동, 배포는 사용자 명시 실행으로 남기는 경계가 실제로 동작했다.

## 언제 쓰나

- 커밋이 다수 쌓일 전면 검토·수정 (단발 버그픽스에는 과함).
- 실거래·프로덕션 등 **회귀 비용이 큰 코드** — 4단계(적대적 검증)의 "되돌려서 테스트 실패 확인"이 특히 값어치를 한다.
