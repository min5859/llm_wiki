# MinT — 백만 LoRA 어댑터 학습·서빙 관리형 인프라

## 개요

MinT 는 단일 베이스 모델 위에서 **수백만 개의 LoRA 어댑터** 를 학습·서빙하는 관리형 인프라다. 전체 체크포인트가 아닌 **어댑터 리비전 (adapter revision)** 만을 정책 단위로 이동시켜, 1조 파라미터 규모의 베이스 모델을 다중 정책 워크로드에 실용적으로 적용할 수 있게 한다.

연구 배경: LLM post-training 이 일회성 작업에서 지속적 인프라 워크로드로 진화하면서, 테넌트별 변형 / 제품 브랜치 / 평가 스냅샷 / 롤백 포인트 등 수많은 정책 변형을 관리해야 한다. 기존 방식은 매 변형마다 수백 GB 전체 체크포인트를 이동·적재하므로 1T+ 규모 베이스 모델에선 확장 불가능.

## 핵심 아이디어 — 어댑터 리비전 중심 재설계

- **베이스 모델은 GPU 메모리에 상주** (resident). 정책 행동은 LoRA 어댑터로만 표현
- 전체 체크포인트 병합·이동 없이 어댑터 파일만 교환
- 체크포인트 크기 비교 (4B 모델 기준):

| 형식 | 크기 |
|------|------|
| Full FT | 8.0 GB |
| LoRA rank-32 | 252 MiB |

훈련–서빙의 경계가 "어댑터 리비전" 단위로 재정의된다.

## 세 가지 스케일링 축

### 1. Scale Up — 1조 파라미터 MoE 까지

- Megatron 분산 훈련 기반, 텐서/전문가 병렬 (TP/EP) 지원
- Kimi K2 (1.04T MoE) 까지 검증
- **R3 (Router Replay)**: MoE 의 전문가 라우팅 불일치 (rollout vs train) 를 해결. rollout 시 선택된 전문가 경로를 기록하고, 학습 시 그대로 재현

### 2. Scale Down — 시간 분할 다정책 학습

- 학습 완료 후 PEFT 포맷으로 어댑터만 export. 옵티마이저 상태 / 랭크별 학습 파일 제외
- 하나의 베이스 할당 하에 **여러 정책을 time-sliced 로 순차 학습**. 유휴 구간을 다른 정책 학습에 재활용

### 3. Scale Out — 3계층 캐시로 10⁶ 정책 주소화

```
정책 카탈로그 (제어 플레인, 주소화 가능 10⁶)
       ↓
CPU 캐시 (정책 메타·텐서 일부)
       ↓
GPU 배치 (실제 활성 정책)
```

- 주소화 가능 정책 수와 GPU 상주 수를 **분리**
- MoE LoRA 텐서를 **패킹**: 37,248개 텐서 객체 → 672개 (cold 로딩 8.5–8.7× 가속)
- 단일 엔진에서 100K 정책 측정, 시스템 전체로는 10⁶ 까지

## 한계

1. **DSA (동적 희소 어텐션) 재현 불완전**: GLM-5 계열의 DSA 인덱서 선택을 토큰 수준으로 완전히 재현하지 못함. IcePop (확률 비율이 신뢰 구간 벗어난 토큰에 가중치 0) 으로 완화만 함
2. **Cold 로딩 직렬화**: 서로 다른 어댑터의 cache miss 요청이 병렬화되지 않고 ~1.35–1.40s/정책 순차 처리 (16개 어댑터 → 23초 계단)
3. **rank-1 제한**: Scale Out 측정은 rank-1 MoE 어댑터 기준. 고랭크 확장성은 별도 검증 필요
4. **weak-locality p95 63s**: 라우팅 정책 미흡 시 지연 폭증

## 실무 적용 가능성

- **다중 테넌트 LLM SaaS**: 사용자별 개인화 어댑터를 단일 베이스 모델 위에서 운영
- **지속 학습 파이프라인**: rollout → update → deploy 사이클 자동화 + 버전 관리 (롤백 포인트) 구조적 지원
- **비용 효율**: 전체 체크포인트 저장 비용 최대 99% 절감 (rank-1 어댑터 ~0.10% 크기), 공유 베이스 모델로 GPU 메모리 중복 제거
- **연구 재현성**: Tinker 호환 API + mint-cookbook 레시피로 SFT/DPO/GRPO 재현 가능

## 시사점

LoRA 자체는 2022년의 기반 기술이지만, **"전체 체크포인트가 아니라 어댑터를 1급 정책 단위로 삼는다"** 는 운영 관점의 재설계가 늦게나마 도구화된 것이 핵심. 1인 개발자 관점에선 직접 구축 대상은 아니지만,

- 다중 LLM provider 어댑터 추상화 (`[[multi-llm-provider-adapter-pattern]]`) 와 비교해, **여러 정책이 같은 베이스 모델을 공유** 한다는 점이 미래 SaaS 형태의 LLM 비용 모델을 결정할 가능성
- post-training 을 "체크포인트 운영" 에서 "어댑터 카탈로그 운영" 으로 보는 멘탈 모델은 추후 RLHF / DPO / GRPO 워크플로 설계 시 참고 가치
