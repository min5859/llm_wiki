# AI 밸류에이션 신뢰도 — 환각·가짜 데이터·수식 결함의 5가지 안티 패턴

LLM 으로 재무 데이터를 처리해 기업가치를 평가하려는 시스템에서 반복적으로 등장하는 5가지 신뢰도 함정. M&A·투자 의사결정에 쓰이려면 이 다섯을 하나라도 남겨두면 안 된다 (출력 숫자가 그럴듯해 보여서 사용자가 검증 없이 인용해 버린다).

## 개요

"AI 가 PDF/JSON 을 읽고 EBITDA·DCF 로 기업가치를 평가" 라는 워크플로우는 다음 5가지 결함을 거의 항상 동시에 가진다.

1. **AI 환각 기반 valuation** — 할인율·성장률·멀티플을 LLM 에게 지어내게 함.
2. **수식 결함** — Net Debt / EV / EBITDA 의 정의 자체가 틀림.
3. **단위 강제 부재** — 원/백만원/억원이 섞인 인풋을 AI 단위 변환에 위임.
4. **빈 인풋 → 가짜 데이터** — null 과 0 을 구분하지 못해 "데이터 없음" 이 "0 원" 으로 해석됨. 폴백 상수가 곳곳에 박혀 있음.
5. **PDF 침묵 실패** — 스캔 PDF 를 빈 텍스트로 받고도 에러 없이 진행, AI 가 백지 입력으로 전체 재무를 날조.

## 안티 패턴 1: AI 환각 기반 valuation

### 증상

```ts
// /api/valuation
const prompt = `다음 재무데이터로 EBITDA/DCF 평가해줘:\n${JSON.stringify(financials)}`;
const result = await ai.complete(prompt);
return parseValuationJSON(result);  // 할인율·성장률·멀티플이 모두 LLM 출력
```

LLM 출력에 "WACC 10%, 성장률 3%, EV/EBITDA 멀티플 8x" 라는 숫자가 그럴듯하게 적혀 있지만, **이 숫자들의 출처는 LLM 의 학습 분포 평균** 이지 해당 기업 / 업종 / 시기에 적합한 값이 아니다. 같은 입력을 두 번 넣어도 다른 valuation 이 나온다.

### 수정 방향

LLM 의 역할을 "**제안자**" 로 한정하고 계산은 결정론적 수식으로:

```
1. 데이터 입력
2. LLM: 가정값 제안 (WACC 후보 / 성장률 후보 / 멀티플 후보, 각각 근거 출처 포함)
3. 사용자: 가정값 확인·수정 (UI 에서 슬라이더 / 입력 필드)
4. 코드: 확정된 가정값으로 결정론적 DCF / EV-EBITDA 계산
5. 출력: 결과 + 가정값 + 데이터 출처 모두 함께
```

이렇게 하면 동일 입력 + 동일 가정값 → 동일 결과 (재현 가능). LLM 이 추정에 관여한 부분을 사용자가 식별·수정할 수 있다.

## 안티 패턴 2: 수식 자체의 결함

`multiples.ts` / DCF 구현에서 흔한 4가지 오류:

| 항목 | 흔한 오류 | 정확한 정의 |
|---|---|---|
| **Net Debt** | `총부채` (현금 미차감) | `차입금 - 현금성자산`. 무이자 부채 (매입채무 등) 는 제외. |
| **EV** | PER/PBR (주주가치) 에 총부채 더함 | EV = 시가총액 + Net Debt + 우선주 + 비지배지분. PER/PBR 는 EV 가 아니라 별도 multiple 라인으로 분리해야 함. |
| **EBITDA** | `영업이익 × 1.3` (감가상각비 미추출) | `영업이익 + 감가상각비 + 무형자산상각비`. 업종별 D&A 비중이 다른데 1.3 일률 적용은 위험. |
| **DCF terminal value** | `WACC ≤ 성장률` 가드 부재 | Gordon growth 분모 (`WACC - g`) 가 0 또는 음수 → 무한대 / 음수 valuation. 입력 검증 필수. |

### Net Debt 차이가 실제로 얼마나 큰가

| 회사 | 총부채 | 차입금 | 현금성자산 | "잘못된 Net Debt" | 정확한 Net Debt |
|---|---|---|---|---|---|
| 캐시 풍부한 IT | 5,000억 | 1,000억 | 8,000억 | **+5,000억** | **-7,000억** (Net Cash) |
| 부채 많은 제조업 | 3조 | 2조 | 5,000억 | **+3조** | **+1.5조** |

Net Debt 부호와 규모가 통째로 바뀐다 → EV 도 통째로 바뀌고 → EV/EBITDA 멀티플 valuation 이 뒤집힌다. **이걸 LLM 에 위임하면 안 된다 — 코드로 정의를 강제**.

## 안티 패턴 3: 단위 강제 부재

### 증상

- `finance_format.json` 같은 템플릿에 "단위: 억원" 이라고 **주석만** 있음.
- 추출 후 단위가 일관됐는지 검증하는 코드가 없음.
- DART API (원), 감사보고서 PDF (백만원 / 천원), 사용자 직접 입력 (억원) 이 섞여 들어옴.

LLM 이 단위를 헷갈리면 1,000배 ~ 1억배 오차가 나는데, valuation 결과는 여전히 그럴듯한 숫자로 보인다 → **검출이 매우 어렵다**.

### 수정 방향

1. **추출 JSON 에 `unit` 메타데이터** 필수:
   ```json
   {
     "revenue": { "value": 5234, "unit_raw": "억원", "unit_normalized": "억원" },
     "operating_income": { "value": 423, "unit_raw": "억원", "unit_normalized": "억원" }
   }
   ```
2. **회계 항등식 자동 검증**:
   - `총자산 = 총부채 + 자본총계` (오차 < 1%)
   - `매출총이익 = 매출액 - 매출원가`
   - `영업이익 = 매출총이익 - 판매관리비`
3. **출처별 단위 결정론적 변환** — DART 는 원으로 확정 → 코드 레벨에서 억원으로 변환 (AI 위임 금지).

## 안티 패턴 4: 빈 인풋 → 가짜 데이터 (5곳에서 동시 발생)

가장 위험한 패턴. **여러 레이어가 각자 폴백을 두고 있어** 어디서 가짜가 만들어졌는지 추적이 거의 불가능해진다.

| 위치 | 안티 패턴 | 결과 |
|---|---|---|
| **프롬프트** | "빈 필드는 최대한 데이터에서 찾아서 채우세요" | AI 에게 **명시적으로 날조를 지시** |
| **DataLoader** | `value ?? 0` (모든 null → 0 변환) | "데이터 없음" 과 "실제 0" 구분 불가 |
| **DCF 폼** | `baseFcf = inputFcf || 100` | FCF 가 0 이면 100 억원 하드코딩 |
| **multiples 계산** | `ebitda = inputEbitda || (operatingIncome * 1.3)` | 데이터 누락이 자동 추정으로 대체, 사용자에게 안 알림 |
| **템플릿 파일** | `finance_format.json` 의 예시값으로 **실제 회사 데이터** 사용 | AI 가 추출 실패 시 다른 회사의 실제값을 그대로 출력 |

마지막 항목 (예시값으로 실제 회사 데이터) 이 가장 잘 안 보이는 함정. 다음과 같이 노출된다:

```json
// finance_format.json (BAD)
{
  "_example": {
    "company": "풍전비철",
    "revenue": 1234,  ← 실제 풍전비철 매출
    "operating_income": 56  ← 실제 풍전비철 영업이익
  }
}
```

AI 가 추출에 실패해 빈 결과를 줄 때 시스템이 이 예시값으로 폴백하면, 사용자는 자기가 분석하려던 회사의 결과로 풍전비철의 실제값을 본다 → **가장 디버깅하기 어려운 silent contamination**.

### 수정 방향

1. **null ↔ 0 구분 복원**:
   ```ts
   // BAD
   const value = data.revenue ?? 0;

   // GOOD
   const value = data.revenue;  // null 그대로 유지
   if (value == null) return { status: 'no-data' };
   ```
2. **모든 폴백 상수 제거** — `|| 100`, `|| (영업이익 × 1.3)` 같은 것들.
3. **프롬프트 변경**:
   ```
   - 없는 데이터는 반드시 null 로 표시
   - 절대 추정·외삽·다른 데이터로 채우지 말 것
   - 단위는 원본 그대로 unit 필드에 명시
   ```
4. **템플릿의 실제 데이터 → placeholder 문자열**:
   ```json
   "_example": {
     "company": "[회사명]",
     "revenue": "[매출액 억원]"
   }
   ```
5. **데이터 완성도 스코어** UI 노출 — "핵심 필드 32개 중 28개 (88%) 가 실제 데이터, 4개 누락".
6. **밸류에이션 시작 전 프리플라이트 체크** — 핵심 데이터 누락 시 valuation 실행 차단 + 경고.

## 안티 패턴 5: PDF 침묵 실패

스캔 PDF 를 빈 텍스트로 추출하고도 에러 없이 진행 → AI 가 백지 입력으로 전체 재무를 날조. 일반 패턴은 [[pdf-text-extraction-vs-ocr]] 참고.

요점:
- `pdf-parse` / `pdftotext` 는 **텍스트 레이어만 추출** → 스캔 PDF 는 빈 문자열 반환.
- 빈 텍스트 감지 시 **반드시 에러로 차단** + "OCR 필요" 사용자 안내.
- AI 추출 호출 전에 입력 길이 검증 (너무 짧으면 차단).

## 데이터 신뢰도 표시 — UI 의무

위 5가지를 모두 수정해도, 사용자가 결과 숫자를 어떻게 신뢰해야 할지 알 수 없으면 의미가 없다. 모든 valuation 출력에 다음 3가지를 함께 노출:

| 메타정보 | 표시 |
|---|---|
| **데이터 출처** | DART 실제값 / AI 추출값 / 사용자 입력값 / 추정값 — 필드별로 색상 구분 |
| **데이터 완성도** | 핵심 필드 N개 중 M개 (X%) 가 실제 데이터 |
| **신뢰도 등급** | 데이터 완성도 + 사용된 가정값 수에 따라 A / B / C / D 등급 |

이게 있으면 "valuation 결과 1조 2,000억" 옆에 "신뢰도 D 등급 (핵심 데이터 30% 누락)" 가 보여서 사용자가 그 숫자를 회의적으로 다룰 수 있다.

## 종합 — 단계별 수정 우선순위

데이터 신뢰도가 낮은 시스템을 받으면 다음 순서로 풀어야 한다 (역순으로 풀면 효과 없음):

| 단계 | 작업 | 이유 |
|---|---|---|
| **1** | null/0 구분 복원 + 모든 폴백 상수 제거 + 프롬프트에서 추정 금지 | **모든 검증의 기반** — 이게 안 되면 다른 어떤 검증도 의미 없음 |
| **2** | 단위 메타데이터 + 회계 항등식 검증 | 1,000배 오차 잡기 |
| **3** | 수식 수정 (Net Debt / EV / EBITDA / WACC 가드) | 정의 자체가 맞아야 valuation 이 의미 있음 |
| **4** | AI 환각 → "AI 제안 + 사용자 확인 + 결정론 계산" 으로 전환 | 재현 가능성 |
| **5** | OCR 추가 + 빈 입력 차단 | 인풋 단계 가시성 |
| **6** | 데이터 신뢰도 메타정보 UI | 사용자에게 회의권 부여 |

순서를 바꾸면 (예: 1단계 없이 4단계 먼저) 폴백 상수가 살아 있어 "결정론 계산" 이 가짜 데이터로 결정론적 결과를 낸다.

## 일반화 — LLM 으로 정량 분석하는 모든 시스템

본 안티 패턴은 재무 valuation 에 한정되지 않는다. LLM 으로 다음 같은 정량 분석을 하는 모든 시스템에 적용:

- 의료 영상 / 임상 데이터 분석
- 법률 사건 가치 평가
- 부동산 valuation
- 보험 손해사정
- 신용평가

공통 핵심: **LLM 은 데이터 추출·분류·제안에 강하지만 정량 계산에서는 결정론적 코드로 위임해야 한다**. LLM 출력 숫자가 그럴듯하다는 사실 자체가 가장 위험한 신호다.
