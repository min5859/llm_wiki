# 0부터 빌드 vs 오픈소스 포크

이미 비슷한 OSS 가 존재할 때 "직접 처음부터 만들 것인가, 그걸 포크해 고쳐 쓸 것인가"를 가르는 판단 틀. 데모/발표용 토이가 **개인 실사용 도구**로 넘어가는 순간 정답이 뒤집히는 경우가 많다.

## 핵심 — 판단 기준은 "맥락"에 따라 바뀐다

| 맥락 | 지배적 판단 기준 | 유리한 쪽 |
|---|---|---|
| 해커톤·포트폴리오·발표 | **차별점**(남과 다른 무엇) | 0부터 빌드 (내 손으로 만든 차별 기능을 보여줘야 함) |
| 개인 실사용·사내 내부 도구 | **① 내가 원하는 동작 ② 유지보수 부담 최소화** | 포크 (검증된 코드 재사용) |

함정은 같은 프로젝트가 시간이 지나며 맥락이 바뀐다는 것. "발표가 끝나면 차별점은 무의미해지고", 남는 건 "내가 매일 쓸 때 동작하는가 + 내가 계속 고칠 수 있는가" 뿐이다. **차별점이라는 전제가 사라지면 기존 구현 재사용의 가치가 압도적으로 커진다.** 빌드를 선택했던 근거(차별화) 자체가 증발하므로 재평가가 필요하다.

## 포크가 유리해지는 신호

- 만들려던 기능의 **상당 부분(체감 ~80%+)을 OSS 가 이미 구현**했다. (예: 인증·세션·스트리밍·스케줄링·승인 플로우 같은 "재미없지만 반드시 필요한" 배관)
- 라이선스가 허용적(MIT/Apache)이라 포크·수정·비공개 운영에 제약이 없다.
- 내가 더할 부분이 **얇은 레이어**(UX·연동·정책)로 분리 가능해, 본체 위에 얹기 쉽다.
- 혼자/소수가 **장기 유지보수**해야 한다 → 직접 짠 코드는 전부 내 부채, 포크는 업스트림이 일부를 떠받친다.

## 0부터 빌드가 유리해지는 신호

- 차별점 자체가 결과물(발표·차별화·학습 목적).
- OSS 의 아키텍처가 내 요구와 정면 충돌해 포크 비용 > 재작성 비용.
- 의존성/라이선스/보안상 외부 코드를 들일 수 없다.

## 포크를 택했을 때의 실무 수칙

- **포크의 핵심 가정을 코드로 검증**하라. README·기존 메모의 "X 는 없다/안 된다" 류 가정이 틀린 경우가 잦다 — 실제로 grep/CLI/엔드포인트로 확인. (실측 사례: "Kanban API 없음" 가정이 네이티브 CLI + dashboard 엔드포인트 존재로 뒤집힘 → [[hermes-agent]])
- **내가 더할 레이어가 본체 어디에 끼는지 지점을 먼저 매핑**하라(컴포넌트·라우트·이벤트). 작업량 추정과 충돌 회피의 기준이 된다.
- from-scratch 잔재 코드는 **삭제 대신 `legacy/` 이동 + 사유 보관**(이력 보존, 되돌리기 가능). 방향 전환은 가역적으로.
- 본체 화면을 **실제로 띄워 보기 전에는 추측 기반 재작업을 미뤄라**(헛수고 방지).
