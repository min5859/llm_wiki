# 폴링 주기 vs 봉 단위 — 알고리즘 트레이딩의 시간 정합성

라이브 트레이딩 엔진의 **폴링 주기 (cycle interval) 와 봉 단위 (bar_interval) 가 일치하지 않으면, 폴링을 빠르게 해도 알파가 늘지 않는다.** 직관에 반하는 함정이라 "주기 단축 = 반응성 향상" 으로 착각하기 쉽다.

## 핵심 사실

| 항목 | 의미 |
|------|------|
| **폴링 주기** | 엔진이 깨어나 시그널을 재평가하는 간격 (예: 10분) |
| **봉 단위 (`bar_interval`)** | 전략이 입력으로 받는 가격 봉의 시간 해상도 (예: 1일봉, 5분봉) |
| **시그널 갱신 주기** | 봉이 마감되어 *새* 데이터가 도착하는 간격 — 사실상 `bar_interval` |

폴링 주기 ≠ 시그널 갱신 주기. **시그널은 봉 단위에 묶여 있고, 폴링은 그 봉을 "몇 번 보러 가느냐" 일 뿐이다.**

## 안티패턴: 일봉 + 빠른 폴링

```yaml
# config/trading.yaml
bar_interval: "1d"        # 전략 입력은 일봉
```

```
# launchd
periodic --interval 10    # 10분마다 폴링
```

- 한국장 09:00~15:30 동안 약 **47회 폴링** → 동일한 일봉을 47번 평가.
- 일봉은 마감 후에만 갱신되므로, 장중 47회 폴링 결과는 모두 동일 시그널.
- 폴링을 10분 → 5분으로 줄여도 같은 결과를 94회 보는 것뿐. **알파 0 증가, API 호출만 2배**.
- 첫 사이클에서 진입했다면 나머지 46회는 포지션 체크로 스킵 — 체결 타이밍 개선 효과도 없음.

## 폴링 주기 단축이 의미를 가지려면

폴링과 봉 단위는 **한 쌍으로 같이 내려야** 한다.

| 시나리오 | bar_interval | 폴링 주기 | 결과 |
|---------|------------|---------|------|
| 일봉 + 10분 폴링 | 1d | 10분 | 일 1회 시그널, 47회 중복 평가 |
| 일봉 + 5분 폴링 | 1d | 5분 | 일 1회 시그널, 94회 중복 평가 (의미 없음) |
| **분봉 전환** | 5m / 30m | 5~10분 | 장중 시그널 다회 발생 → 알파 가능 |
| 일봉 + 종가 근처 집중 폴링 | 1d | 14:30~15:20 만 5분 | 종가 근처 진입으로 슬리피지 ↓ |

> 폴링 주기는 *그 봉이 새로 그려질 빈도* 와 비슷하게. "분봉 알파를 잡으려면 분봉을 입력으로 줘야 한다" 가 1차 조건.

## 일봉 유지 시 진짜 효과를 내는 레버

봉 단위는 그대로 두고 싶다면, 폴링 주기 단축이 아니라 **진입/청산 타이밍 자체** 를 손봐야 한다.

| 레버 | 효과 |
|------|------|
| 시초가 진입 회피 | 갭/스파이크 슬리피지 회피 |
| 종가 근처 (14:30~15:20) 진입 | 일봉 시그널을 *그 일봉의 종가에 가깝게* 체결 |
| VWAP/TWAP 분할 매수 | 단일 가격 노출 감소 |
| 장중 1회 폴링 (15:00) | API 호출 ↓, 결과 동일 |

> 일봉 전략에서 "10분마다 깨우기" 의 진짜 가치는 *시그널 갱신* 이 아니라 *주문 상태 추적·재시도·청산 모니터링* 에 있다. 그 가치를 인식하지 않은 채 폴링 주기로 알파를 끌어내려 하면 노력 대비 효과가 없다.

## 예외: 실시간 가격 기반 청산 (트레일링/손절)

위의 "폴링 단축 = 알파 0" 규칙은 **시그널이 봉 종가(`bar.close`) 에 묶인 경우** 에만 성립한다. 청산 규칙이 **실시간 평가가(`pos.current_price`, 증권사 실시간 평가 단가)** 를 기준으로 하면 폴링 주기가 곧 해상도가 되어, 일봉 유지 상태에서도 폴링 단축이 실제 효과를 낸다.

- 트레일링스톱·손절은 봉 종가가 아니라 **포지션의 현재 평가가**로 트리거된다.
- 엔진이 장중 매 사이클 `cache_updated_at=0` 으로 리셋해 포지션을 강제 갱신하면, `current_price` 는 `bar_interval` 과 무관하게 **폴링 주기마다** 새 값으로 갱신된다.
- 따라서 폴링 10분 → 2분 단축은 트레일링스톱의 **고점-청산 해상도를 실제로 높인다** (peak 대비 더 가깝게 청산 → 수익 보존 ↑). 시그널(진입) 알파는 안 늘지만 *청산 타이밍* 은 개선된다.
- 트레이드오프: 사이클당 잔고·시세 호출이 ~5배. 단 KIS rate limit 은 초당 기준이라 2분 사이클은 예산 안에서 안전.

> 정리: **봉 종가 기반 평가(진입 시그널)** 는 폴링 단축 무의미, **실시간 평가가 기반 청산(트레일링/손절)** 은 폴링 단축이 유의미. 같은 엔진 안에서 두 종류가 공존하므로 "폴링을 줄여야 하나?" 는 *무엇을 기준으로 트리거되는가* 로 갈라서 판단해야 한다.

### 후속 교훈: 단일 사이클을 청산용으로 줄이면 매수에 부작용 (2026-06-23)

위 예외(트레일링 해상도를 위해 폴링 10→2분 단축, commit `1bb80f1`)는 **매수·매도가 한 폴링 사이클을 공유한다는 함정**을 안고 있었다. 사이클을 2분으로 줄이자 **매수 평가도 2분마다 돌아**, 미체결·거부되는 매수 주문이 하루 수십 회까지 폭증해 일일 주문 한도를 조기 소진시켰다(트레일링 SELL까지 막힘 — [[risk-control-exemption-and-failed-attempt-accounting]]).

해결책은 **행위별 사이클 분리**: `StrategyContext.allow_buy` 플래그 + `run_periodic(interval, buy_interval)`로 매도/트레일링은 2분, 매수는 30분 주기로 분리(commit `d9e5a44`). 가드를 pending 등록 **전**에 두어 매수 시그널이 전략 상태를 오염시키지 않게 했다.

> 원칙: "폴링 단축이 유의미한 행위(청산)"와 "무의미하거나 유해한 행위(진입)"가 한 루프에 묶여 있다면, 폴링 주기를 손대기 전에 **행위별로 사이클을 분리**하라. 한 행위를 위한 튜닝이 다른 행위의 부작용을 낳는다.

## 진단 체크리스트

새 폴링 주기 또는 새 bar_interval 을 적용하기 전에 확인:

1. `bar_interval` 이 무엇인가? (config 와 코드 양쪽 모두)
2. 한 사이클 (장 시간 / 24시간) 동안 봉이 *몇 번* 갱신되는가?
3. 폴링 주기 ÷ 봉 갱신 간격 = 1 사이클 중복 평가 횟수
4. 중복 평가가 10회 이상이면, 폴링 주기를 더 줄여도 알파에는 영향 없음
5. 줄여야 할 것은 **봉 단위** 인가, **폴링 주기** 인가? 혼동 금지
