# 개인 LLM wiki 큐레이션 — write-only 창고를 피하는 수집·승격 설계

자동 ingest 기반 개인 지식 wiki(v1)를 1년 가까이 운영한 실측 결과에서 도출한 큐레이션 방법론. **wiki의 가치는 쌓인 문서 수가 아니라 답변 시 다시 읽히는 빈도(재조회율)에서 나온다.** v1은 194개 문서 중 절반이 analyses였고 주제가 디스크 정리·논문 스크랩·게임 개발까지 산개해, 다시 읽힌 것은 상위 20~30개뿐인 write-only 창고가 됐다.

## 실패 원인 — 종류가 다른 지식을 한 통에 섞음

지식은 세 종류이고 각각 갈 곳이 다르다:

1. **프로젝트 고유 지식** (코드베이스 구조·결정·버그) → **각 프로젝트 repo 안에** (CLAUDE.md, docs/). 작업 디렉터리 문서는 자동으로 읽히므로 재조회가 공짜. 별도 wiki로 빼면 코드와 떨어져 낡는다.
2. **프로젝트를 넘나드는 반복 지식** (증권 API 함정, launchd 패턴, 토큰 절약법) → **중앙 wiki**. 단, 도메인을 한정해야 한다. 한 프로젝트에 넣으면 다른 프로젝트에서 못 찾는다.
3. **일회성·뉴스성** → 어디에도 저장하지 않는다. v1을 창고로 만든 주범.

## 설계 원칙

- **범위 먼저** — "3개월 뒤 내가 다시 검색할 주제"를 2~3개 도메인으로 못 박고 그 외는 수집 자체를 제외. 이 기준을 vault CLAUDE.md에 명문화하면 자동 ingest가 스킵 판단의 게이트로 쓴다.
- **push보다 pull (2회 등장 승격)** — 세션에 한 번 나온 지식은 소비하고 버리고, 두 번 이상 재등장하거나 실제로 다시 찾아본 것만 승격. 유지비 대비 효과가 훨씬 좋다. (도메인 핵심 버그·결정은 예외적으로 즉시 기록.)
- **읽기 경로를 설계에 포함** — "질문 답변 전 wiki 검색 → 답변에 인용" 루프가 실제로 도는지를 성공 기준으로 삼는다. 이게 없으면 어떤 구조든 다시 창고가 된다.
- **저장과 승격의 2단계 분리** — 세션 로그는 전량 로컬 버퍼에 받아두고(git 제외), 필터는 승격 단계(주기적 ingest)에서 CLAUDE.md 기준으로 작동시킨다. 세션 중 "기록할 가치가 있나"를 판단할 필요가 없어진다. 대신 처리 완료 로그가 무한 증가하므로 보존 정책(ingested 후 N일 삭제)이 필요하다.

## 이관(마이그레이션) 선별 — 감이 아니라 재사용 신호로

구 wiki에서 옮길 문서는 두 가지 데이터로 골랐다: ① **다른 문서가 인용한 횟수** (5회 이상 = 허브 지식), ② **변경 이력 횟수** (2회 이상 = 반복 갱신된 살아있는 지식). 여기에 검색엔진으로 다시 찾기 어려운 도메인 고유 지식(예: 증권사 API의 D+2 결제 거부)을 수동 추가. 결과: 194건 중 73건만 이관, 신호가 높아도 도메인 밖이면 원칙대로 제외.

## v1→v2 재편 사례 (2026-07)

이 문서의 설계 원칙이 실제로 어떻게 발현됐는지의 사후 기록. v1(도메인 `work`/`personal`, 8폴더, 194문서)이 재조회되지 않는 write-only 창고가 됐다는 진단에 따라, v2는 도메인을 `ai-agent`·`trading` 2개로 좁히고 폴더도 6개(concepts/analyses/patterns/projects/decisions/bugs)로 단순화했다.

- **선별 이관 기준** — 감이 아니라 재사용 신호로: ① 다른 문서가 5회 이상 인용 ② 변경 이력 2회 이상 ③ 검색엔진으로 못 찾는 도메인 고유 지식. 결과: 194건 중 **73건(37.6%)** 만 이관.
- **미이관 121건의 처리 원칙** — 삭제하지 않고 "필요해지면 v1에서 개별 복사"를 공식 되돌리기 경로로 남겼다. v1 저장소는 GitHub에 그대로 보존하며, 로컬 정리(아카이브)만 하고 원격은 지우지 않는다 — 이관 기준이 잘못됐을 때의 안전판.
- **문서 갱신 방식의 진화** — v2 초기엔 frontmatter `updated` 필드 갱신이 전부였지만, 운영이 누적되며 각 `## 변경 이력` 줄에 출처 세션 로그 링크와 `[[다른문서]]` 상호 참조를 함께 새기는 방식으로 굳어졌다. 그 결과 "이 갱신이 어느 세션에서 왔고 어떤 문서와 함께 바뀌었는지"를 역추적할 수 있다 — 재조회율을 높이는 부수 효과.
- **운영 교훈 — vault 개명은 절대경로 참조 전수조사가 선행돼야 한다**: `llm_wiki2` → `llm_wiki` 개명 시, vault 밖의 외부 자동화가 절대경로로 vault를 하드코딩 참조하고 있어 일괄 갱신이 필요했다 — gieok LaunchAgent plist **3개**(ingest/lint/log-retention)와 Claude Code `settings.json` 훅 **7곳**. plist는 파일 내용을 고쳐도 `launchctl unload`/`load` 재로드가 **수동 실행 필요**(권한상 자동화 불가)라는 점도 반복 확인됐다 — [[gieok]] 참고.
