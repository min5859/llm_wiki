# 안전장치가 손실을 키울 때 — 리스크 감축 주문 면제와 실패 시도 회계

"트레일링 스톱이 발동했는데 매도가 안 됐다"는 증상에서 출발해, **안전장치(일일 주문 한도)가 오히려 손절·트레일링 매도를 차단해 손실을 키운 사례**에서 도출한 범용 교훈 묶음. 트레이딩 도메인의 사례지만, 외부 자원에 주문/요청을 보내며 자체 처리율 제한을 두는 모든 시스템에 적용된다.

## 1. 리스크 감축 주문은 처리율 제한에서 면제하라

일일 주문 한도·rate limit 같은 안전장치를 **BUY·SELL 무차별로** 적용하면, 한도 소진 후에는 손절·트레일링 매도까지 막혀 **안전장치가 손실을 키우는** 안티패턴이 된다.

```python
# 안티패턴: 한도 게이트가 매수/매도를 구분하지 않음
if order_count >= max_daily_orders:
    break   # SELL(손절·트레일링)도 함께 차단됨

# 수정: 게이트는 진입(BUY)에만, 리스크 감축(SELL)은 면제
if signal.side == "BUY" and order_count >= max_daily_orders:
    continue
```

원칙: **처리율 제한은 진입·확대 행위에만 걸고, 리스크를 줄이는 행위(청산·손절·취소)는 무조건 통과시킨다.** 한도의 목적은 과잉 거래 억제이지 손실 방치가 아니다.

## 2. 실패한 시도는 한도·카운터를 소비하지 않는다

주문 제출이 **실패(거부·None 반환)해도** 카운터를 무조건 증가시키면, 외부 거부가 자체 한도를 갉아먹는 결합 결함이 된다.

```python
# 안티패턴: 성공/실패와 무관하게 카운터 증가
submit_order(...)            # 실패해도 None 반환
self._increment_order_count()   # 무조건 +1

# 수정: 성공(주문 ID 반환) 분기 안에서만 증가
order_id = submit_order(...)
if order_id is not None:
    self._increment_order_count()
```

실제 사례에서는 1주 매수가 거래소에서 수십 회 거부되며 매번 카운터가 올라, **실제 체결되지도 않은 주문이 일일 한도를 조기 소진**시켰다. 거부 시에는 카운터 증가 대신 **백오프(재시도 억제)** 훅을 두는 것이 옳다.

## 3. "포기"와 "취소"는 다르다 — 유령 체결

지정가 미체결을 "만료" 처리할 때 **내부 추적(pending)만 삭제하고 거래소의 실제 주문을 취소하지 않으면**, 거래소에 OPEN으로 남은 주문이 나중에 체결되어 내부 상태와 실제 잔고가 어긋난다(유령 체결).

원칙: **외부에 상태를 만든 행위(주문 제출)를 포기할 때는 반드시 외부에 취소/정리 요청을 동반**한다. 내부 dict에서 키를 지우는 것은 "포기"일 뿐 "취소"가 아니다.

구현 팁: 전략이 broker를 직접 호출하지 않고 **취소를 큐잉 → 엔진이 drain**하면 전략의 broker 비의존성과 테스트 용이성을 유지하면서 실제 취소를 보장할 수 있다.

## 4. startup 전용 reconciliation은 장기 가동의 구멍

내부 상태 ↔ 외부 잔고 동기화(reconciliation)를 **시작 시 1회만** 수행하면, 무재시작으로 며칠씩 도는 라이브 프로세스에서는 갭이 생긴다. 취소 실패·시세 캐시 지연·재시작 공백을 모두 포괄하려면 **멱등(idempotent) 동기화 메서드를 주기적으로** 호출하는 안전망이 필요하다.

## 5. 가용 현금 ≠ 예수금

`get_cash()`가 예수금만 반환하고 **미체결 매수가 묶어둔 reserved 금액을 차감하지 않으면**, 시스템은 "현금 있다"고 보고 제출하지만 거래소는 `주문가능금액 초과`로 거부한다 → 제출-거부 루프. 가용 현금은 `예수금 − 미체결 매수 예약금`으로 계산해야 한다.

## 6. 자체 한도 vs 외부 거부를 먼저 가른다 (진단법)

차단 원인을 진단할 때 가장 먼저 **"한도가 누구 것인가"**를 가른다.

- **자체 카운터 차단**: 우리 코드의 게이트가 막음 (거래소는 그 시그널을 본 적도 없음)
- **외부(거래소) 거부**: 시그널이 거래소까지 갔으나 거부됨 (`주문가능금액 초과` 등)

이번 사례의 SELL 미발생은 **100% 자체 카운터 차단**이었다 — 거래소 로그에는 SELL이 전혀 없었다.

## 7. 음성 증거(이벤트 부재) 기반 진단

"주문 제출: SELL" 로그가 **0건**이라는 사실 자체가, 시그널 생성과 브로커 전송 사이가 끊겼다는 결정적 단서였다. 침묵 결함이 손실로 직결되는 도메인에서는 **있어야 할 이벤트가 없음**을 적극적으로 검증해야 한다(`grep ... | wc -l` 이 0인지 확인).

## 단일 폴링 사이클의 부작용

이 사건의 증폭 요인은 트레일링 해상도를 높이려 폴링 주기를 단축한 것이었다. 매수·매도가 한 사이클을 공유하면, 청산을 위한 주기 단축이 매수 거부 폭증을 부른다. 행위별 사이클 분리가 답이며, 자세한 것은 [[polling-interval-vs-bar-interval]] 참조.
