# 벤치마크 존재 시 절대손절 elif dead code — 손절 자체가 꺼진다

ht_trading 의 Rule1 손절이 `if benchmark: 상대손절 elif: 절대손절` 구조라, 라이브에서는 벤치마크(069500 KODEX200)가 항상 붙어 **절대손절(-10%)이 도달 불가능한 dead code** 였던 버그. 지수 동반 하락 시 상대 조건이 안 걸리고, `stale_holding_days=0` 이라 다른 바닥도 없어 **개별 종목의 하방이 사실상 무제한**이었다. 실측으로 -19% 종목을 계속 보유 중이었다.

## 증상 → 근본 원인

- 증상: 절대손절 기준(-10%)을 훨씬 지난 종목이 청산되지 않고 보유 지속.
- 근본 원인: 상대손절(지수 대비)과 절대손절이 `if/elif` 로 배타 결합. 라이브에선 벤치마크 시세가 항상 존재하므로 `elif` 분기(절대손절)는 영원히 실행되지 않는다. "상대손절로 완화"라는 의도가 "손절 전체 OFF"라는 결과로 조합된 사례.

## 수정 — 3층 손절 체계 (2026-07-04, commit 07e473b)

| 층 | 규칙 | 의도 |
|---|------|------|
| Rule1a | **절대 백스톱 -20%** — 벤치마크 무관, 항상 먼저 검사 | 상폐·미회복형 재난 방어 (무조건 가드) |
| Rule1b | 상대손절 -10% — 지수 대비 & 절대 동시 충족 시 | 지수 동반 하락(패닉장)에선 미발동 → 패닉매도 회피 |

백스톱(-20%)을 상대 min-loss(-10%)보다 깊게 잡아 "지수와 같이 빠질 땐 팔지 않는다"는 원래 의도를 보존하면서, 그 아래로는 무조건 잘리게 했다.

## 백스톱 레벨은 스윕 최적값이 아니라 tail 보험으로 정한다

2024~25 백테스트(34종목, 슬리피지 반영) 스윕:

```
백스톱 없음  19.07%  /  -20%  18.13%  /  -15%  16.69%  /  -10%  15.31%
```

V자 회복장 표본에서는 백스톱이 항상 수익률을 깎는다(-20% 기준 연 ~0.9%p). 그러나 이 유니버스는 **생존 편향**(상폐·미회복 종목이 표본에 없음) 하에 있으므로, 스윕 수치로 "백스톱 무용" 결론을 내면 안 된다. -20% 는 스윕 최적값이 아니라 **tail 이벤트 대비 재해보험**(보험료 연 ~1%p 미만)으로 사용자가 선택한 값.

## 일반 교훈

1. **벤치마크(외부 데이터) 의존 로직의 fallback 으로 무조건 가드를 두지 말 것** — 외부 데이터가 "항상 있다"면 fallback 은 dead code 다. 무조건 가드(백스톱)는 별도 층으로 항상 먼저 검사한다.
2. **손절 완화 옵션들이 조합되면 손절 전체가 꺼질 수 있다** — 상대화(완화) + elif 배치(배타) + stale 비활성의 3중 조합이 하방 무제한을 만들었다.
3. **생존 편향 유니버스에서 백스톱 파라미터를 백테스트로 최적화하지 말 것** — 백스톱이 지키려는 시나리오 자체가 표본에 없다.

## 페이퍼 미러링 (2026-07-04, ht_dde)

페이퍼(리웨이트)는 원래 -10% 무조건 손절이라 실거래와 괴리 — 실거래가 -19% 종목을 보유 중인데 페이퍼였다면 잘렸을 것이며, 이 괴리가 하락장 손익곡선 분기의 핵심 원인이었다. [[ht-dde]] 리웨이트가 이 3층 체계를 미러링했고, 미러 드리프트는 [[mirror-config-drift-guard-test]] 테스트로 감시한다. 매수 리듬 괴리(실거래 매수 30분/매도 2분 + 3분할 vs 페이퍼 2분 일괄)도 함께 확인됨.
