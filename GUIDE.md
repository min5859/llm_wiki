# LLM Wiki 작성 가이드

Karpathy의 LLM Wiki 패턴을 기반으로 한 개인 지식 축적 시스템 사용법입니다.

---

## 핵심 개념

기존 방식의 문제: AI에게 같은 주제를 반복해서 물으면, AI는 매번 처음부터 다시 생각합니다. 지식이 쌓이지 않습니다.

LLM Wiki 방식: **AI가 위키를 직접 작성하고 유지**합니다. 새 자료가 들어오면 AI가 관련 위키 페이지들을 업데이트하고, 다음번엔 축적된 위키에서 바로 답합니다.

```
Raw sources (불변) → Wiki (AI가 유지) → 질문/답변
```

---

## 일상적인 사용법

### 새 자료를 추가할 때

```
"office/raw/ 에 이 회의록 저장하고 wiki 업데이트해줘"
"이 논문 내용을 wiki에 추가해줘"
"이 링크 내용 읽고 관련 wiki 페이지들 갱신해줘"
```

Claude Code가 자동으로:
1. `raw/`에 원본 저장 (또는 이미 있는지 확인)
2. 관련 wiki 페이지 검색
3. 기존 페이지 업데이트 또는 새 페이지 생성
4. frontmatter의 `sources`, `updated` 갱신

### 지식을 조회할 때

```
"wiki에서 ○○에 대해 알고 있는 것 알려줘"
"○○ 관련해서 우리 wiki에 뭐가 있어?"
```

Claude Code가 wiki를 먼저 검색하고 답합니다.

### 지식을 정리할 때 (주기적으로)

```
"wiki 전체 훑어보고 오래된 정보나 빠진 연결고리 알려줘"
"confidence가 low인 항목들 목록 뽑아줘"
```

## 도메인 구성 예시

처음에는 하나의 도메인으로 시작하고, 필요해지면 나눕니다.

```
llm_wiki/
├── office/           # 업무 관련
│   ├── raw/
│   └── wiki/
├── research/         # 공부·리서치
│   ├── raw/
│   └── wiki/
└── personal/         # 개인 관심사
    ├── raw/
    └── wiki/
```

**시작은 좁게**: 처음에 도메인을 너무 많이 나누지 마세요. `office/` 하나로 시작해도 충분합니다.

## Raw 자료 형식

`raw/`에는 무엇이든 저장할 수 있습니다.

| 자료 유형 | 저장 방법 |
|---|---|
| 웹 페이지 | URL을 Claude에게 주면 내용을 마크다운으로 저장 |
| PDF / 문서 | 파일 직접 복사 |
| 회의록 | 텍스트 파일로 저장 |
| 메모 | 마크다운 파일로 저장 |
| 이미지 | 파일 복사 (Claude가 내용 분석 후 wiki에 기록) |

파일명 규칙: `YYYY-MM-DD-주제.확장자`
예: `2026-04-14-karpathy-llm-wiki.md`

## Wiki 문서 작성 원칙

Claude Code가 wiki를 작성할 때 지켜야 할 원칙이지만, 직접 작성할 때도 동일하게 적용합니다.

**1. 사실 중심으로**
의견·추측은 명확히 구분합니다. 출처가 없는 내용은 `confidence: low`로 표시합니다.

**2. 짧고 밀도 있게**
한 페이지는 한 주제. 여러 주제가 섞이면 분리합니다.

**3. 연결하기**
다른 wiki 페이지와 연결고리를 `related` 필드와 본문 내 링크로 표시합니다.

**4. 변경 이력 유지**
무엇이 언제 왜 바뀌었는지 `## 변경 이력`에 한 줄씩 기록합니다.

## 실제 예시: 새 자료 추가 흐름

1. **사용자**: "이 논문 읽고 wiki에 추가해줘" → 파일 첨부 또는 링크 제공

2. **Claude Code**:
   - `raw/research/2026-04-14-논문제목.md` 저장
   - `wiki/research/` 검색 → 관련 페이지 확인
   - 기존 `llm-patterns.md` 업데이트 + 새 `paper-논문제목.md` 생성
   - "3개 페이지를 업데이트했습니다. 확인해주세요." 보고

3. **사용자**: "llm-patterns.md 에 추가한 내용 보여줘" → 확인 후 승인

## 자주 쓰는 명령 패턴

```
# 자료 추가
"[URL/파일] raw에 저장하고 관련 wiki 업데이트해줘"

# 주제 조회  
"wiki에서 [주제] 관련 문서 모두 보여줘"

# 새 주제 시작
"[주제]에 대한 wiki 페이지 새로 만들어줘. 지금 아는 것부터"

# 정기 정리 (월 1회 권장)
"wiki 전체 점검: 중복, 오래된 정보, confidence low 항목 리포트해줘"

# 연결고리 찾기
"[주제A]와 [주제B] 사이에 연결할 수 있는 wiki 있어?"
```

## 주의사항

- **raw는 절대 수정하지 않습니다.** 원본 자료의 무결성이 신뢰의 근거입니다.
- **wiki는 AI가 틀릴 수 있습니다.** confidence 필드를 확인하고, `low`는 반드시 검증하세요.
- **너무 많이 나누지 마세요.** 처음 3~5개 자료로 시작해서 자연스럽게 확장합니다.
- **완벽함보다 지속성.** 불완전한 wiki라도 지속적으로 업데이트되는 것이 빈 wiki보다 낫습니다.
