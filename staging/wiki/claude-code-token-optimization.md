# Claude Code 토큰 최적화 — 소비 절반 줄이기

1200시간 이상 Claude Code를 운영한 실전 데이터를 바탕으로 토큰 소비 구조와 절약 전략을 정리한다.

## 핵심 내용

### 토큰 소비 4계층 구조

| 계층 | 구성 | 특징 |
|---|---|---|
| 1. System Prompt | Claude Code 내장 지시 | 매 요청 고정 포함, 수정 불가 |
| 2. CLAUDE.md | 프로젝트 지시서 | 매 요청마다 포함, 길수록 비용 증가 |
| 3. Context Window | 대화 이력 전체 | 누적 증가, 세션이 길수록 폭발적 증가 |
| 4. Tool Results | 파일 읽기·실행 결과 | 파일 크기에 비례 |

> 실전 교훈: 토큰 소비의 대부분은 누적된 대화 이력(계층 3)에서 발생한다.

### Context Rot 문제

컨텍스트가 길어질수록 Claude의 응답 품질이 저하되는 현상.

- 앞부분 내용을 "잊어버리거나" 혼동
- 불필요한 정보가 새 판단을 방해
- 응답 속도 저하 + 비용 증가

해결책: 주기적인 `/compact` 또는 `/clear`

### /compact vs /clear 전략

| 상황 | 권장 커맨드 |
|---|---|
| 작업 중간, 맥락 유지 필요 | `/compact` |
| 작업 완료, 새 작업 시작 | `/clear` |
| 컨텍스트 95%+ 도달 | `/clear` (강제) |
| 컨텍스트 80-94% | `/compact` |

**`/compact` 효과**: 최대 77% 토큰 회복, 중요 맥락 보존

## 세부 사항

### 실전 토큰 절약 전략

#### 1. CLAUDE.md 최적화

- CLAUDE.md가 길수록 매 요청마다 비용 증가
- 불필요한 내용 제거, 핵심만 유지
- 200줄 이상이면 분리 고려

#### 2. 파일 읽기 최소화

- 필요한 파일만 선택적으로 읽기
- 큰 파일은 특정 섹션만 읽기 (offset/limit 활용)
- 전체 코드베이스를 무분별하게 읽게 하지 않기

#### 3. 세션 분리

- 독립적인 작업은 `/clear` 후 새 세션으로 시작
- 하나의 세션에 모든 것을 몰아넣지 않기

#### 4. /plan 모드 활용

- 복잡한 작업 전 계획 검토 → 계획 승인 시 컨텍스트 자동 초기화
- 잘못된 방향으로 진행하다 되돌리는 비용 절감

#### 5. Tool 결과 최소화

- 테스트 실행 시 전체 로그 대신 요약만 요청
- 파일 내용 대신 라인 번호·함수명만 참조하게 하기

### 비용 기준점 (참고)

- Claude Sonnet 기준: 입력 $3/M tokens, 출력 $15/M tokens
- 1회 세션 평균: 수백만 ~ 수천만 tokens (작업 규모에 따라)
- `/compact` 1회로 세션 비용 20-40% 절감 가능

### 컨텍스트 사용률 모니터링

```bash
/context
```

출력 예시:
```
Context: 68% used (136k / 200k tokens)
```

| 수치 | 상태 | 조치 |
|---|---|---|
| ~50% | 정상 | 계속 진행 |
| 50-79% | 주의 | 모니터링 |
| 80%+ | 경고 | `/compact` |
| 95%+ | 위험 | `/clear` |

## 관련 맥락

- 1200시간 운영 데이터: CLAUDE.md 품질이 토큰 효율성에 가장 큰 영향
- 팀 단위 사용 시 개인별 사용 패턴 차이가 비용 차이로 이어짐
- Max 20× 플랜($200/월)이 일반 API 사용보다 고비용 작업에서 경제적인 경우 있음
