### OpenClaw/Hermes vs Codex/Claude Code 용도 구분
대화에서 정리된 실용 구분법:

| 케이스 | 도구 |
|---|---|
| 정해진 워크플로우 자동화, 스케줄링, 폐루프(closed-loop) 반복 작업 | OpenClaw / Hermes |
| 큰 구조 변경, 설계, 검증 | Claude Code |
| 새 잡 개발, 하네스 구축 | Codex + Claude Code 병행 |
| 모바일+PC 연결 업무, MCP 연동 | Codex 앱 |

- "폐루프는 오픈클로에게, 폐루프가 아닌 건 클코나 코덱스" — P28 요약
- OpenClaw 의 실제 활용 예: 슬랙 한마디로 어드민 로그인·조치 자동 트리거

### AI 에이전트 장기 메모리 현황 (2026-06 기준)
- 단순 파일 요약 누적 방식은 파일이 거대해져 컨텍스트 낭비 문제 발생
- 벡터 임베딩 + 시맨틱 검색 방식(honcho, mem0 등)이 등장했으나 **recall 타이밍 제어가 아직 미성숙** — "어떤 걸 언제 가져올지가 제대로 작동하지 않는다"(P86)
- RAG 가 어려운 핵심 이유: QA 루프를 충분히 돌려야 환각이 줄어드는데, 결국 QA 는 사람이 해야 함 → 일반 개인 사용자 수준에서는 노력 대비 리턴이 아직 낮음
- Codex/Claude Code 내장 메모리는 사실상 `MEMORY.md` 파일 하나에 기록 후 필요 시 검색하는 수준
- "AGI 도달을 위한 난제 중 하나" — P86

### LLM Wiki vs RAG 접근
- LLM Wiki 거절파: SQL로 직접 벡터 검색, 정통 RAG 방식 선호
- LLM Wiki 지지파: Obsidian 으로 구현, 연결 관점의 온톨로지
- "도달점은 같고 방법의 차이" — P27
- 실패 사례는 공유되지 않아 조용한 편, 성공 사례만 커뮤니티에 노출되는 편향 존재

### 엑셀 캡처 → 파일 변환 팁
- 스크롤하며 찍을 때 **1개 이상의 row 가 겹치도록** 촬영
- "좌측 첫 컬럼은 row number" 임을 AI 에게 명시 → 이어 붙이기 참조점으로 활용
- 클로드에 이미지 10장 입력 → 약 8분 내 엑셀 변환, 정확도 체감 약 90%
- 결과물 평가: 검토를 3회 이상 요청하거나, 오타·개선 방향을 사람이 피드백 후 재검수 루프

### 오픈라우터 모델 가격 비교 (언급 기준)
- 미니맥스: 인풋 기준 가장 저렴, 단점은 성능 낮음
- DeepSeek: 아웃풋 기준 저렴, "1달러 쓰는 데 한참 걸린다"
- Kimi: 가성비 상대적으로 낮다는 평
- GLM 5.2: 성능은 좋다는 평 (P113)
- Claude: 오픈라우터 경유 시 가격 높음
