# AI 요약

## 인사이트·노하우

- **OpenClaw 크론탭 토큰 절약 전략**: OpenClaw 내장 크론을 리눅스 기본 크론탭으로 대체하면 토큰 소비를 크게 줄일 수 있다.
  - 원칙: "AI가 판단해야 하는 작업만 OpenClaw 크론, 단순 스크립트 실행·크롤링·파싱·계산은 리눅스 크론탭"
  - 하트비트(heartbeat) 크론도 필요 없으면 비워두는 것이 절약에 효과적
  - OpenClaw는 대시보드(관리 UI) 역할로만 남기는 방식

- **Fable 모델이 조용히 Opus로 라우팅되는 문제**:
  - 현상: Fable Ultra Code로 세션을 돌려도 사용량이 줄지 않음
  - 원인: Claude Code의 자동 모델 라우팅이 내부적으로 Opus로 전환
  - 확인 방법: `/btw 너 지금 페이블로 도는거 맞냐` 같이 현재 모델을 직접 질의
  - 대응: 초반부터 Fable이 하위 작업을 위임하도록 명시적으로 지시하면 알아서 나눔. Claude Code 설정에서 "메시지 감지 시 모델 전환(폴백)" 옵션 확인 가능하나, Fable의 강제 라우팅 구조 때문에 완전히 막기는 어려움

- **OpenClaw 크론탭 내 평문 암호 → 암호화 일괄 처리**: 평문으로 저장된 크론탭 스크립트 10개를 OpenClaw에게 한 번에 암호화 수정 지시 → 오류 없이 완료. 토큰 38% 소비했으나 반복 작업 일괄 처리에는 유효한 방식.

- **로컬 영상 컷편집 워크플로 (Claude Code + 로컬 모델)**:
  - 스킬 없이도 컷편집 자체는 동작함
  - `video-use` 검토 후 ElevenLabs API 전사(transcription) 대신 **Whisper 로컬 모델** 방식으로 수정 지시
  - Fable 또는 Opus 모두 가능, 클로드 코드 토큰만 소비, 영상 처리 비용은 무료
  - 확장 가능 방향: 이미지 타이틀 생성(힉스필드 MCP 활용), 배경음악 삽입, 최종 렌더링까지 자동화 가능성 확인됨

- **OpenClaw 반복 작업 시 컨텍스트 리마인드 패턴**: 새 세션 시작 시 기존 작업 내용을 먼저 읽히고("기존작업 읽어라") 내용을 추가하는 방식으로 연속성 유지 — "정독, 흡수, 검증" 자연어 명령 활용

## 반응이 컸던 화제

- **Fable → Opus 자동 라우팅 문제**: Fable로 돌리는 줄 알았는데 실제로는 Opus가 돌고 있었다는 경험 공유 → 여러 명이 `/btw`로 확인하는 방법, 폴백 설정 위치, 강제 라우팅 구조에 대한 설명으로 이어짐. "이렇게 조용히 풀리는지 몰랐다"는 공감이 화제의 핵심.

- **로컬 영상 컷편집 자동화 성공 사례**: 여행 영상을 Fable로 컷편집·전사·배경음악·렌더링까지 자동화한 사례 공유 → 사용 스킬, 비용, OS 요건 등 후속 질문 다수 발생.

## 자동화·시스템 적용 후보

- **AI 판단 유무에 따른 크론 분리 패턴**: AI 추론이 필요한 작업 → OpenClaw 크론 / 단순 실행·수집·계산 작업 → 시스템 크론탭. 토큰 비용 최적화를 위한 분기 기준으로 범용 적용 가능.

- **로컬 Whisper + Claude Code 영상 편집 파이프라인**: ElevenLabs 등 외부 API 대신 Whisper 로컬 모델로 전사 → Claude Code가 컷편집·자막·렌더링 자동화. 비용 없이 영상 후반작업 자동화 가능한 워크플로.

- **Astryx Agent-ready 디자인 시스템 + MCP 연동**: CLI/MCP를 통해 AI 에이전트가 직접 UI 스캐폴딩·테마 생성·컴포넌트 탐색을 수행. Claude Code 또는 Cursor 기반 프론트엔드 자동화 파이프라인에 통합 후보.

무엇인가:
Astryx는 Meta가 공개한 오픈소스 디자인 시스템 페이지입니다. 핵심 메시지는 “완전히 커스터마이즈 가능하고, AI 에이전트가 사용할 수 있는 디자인 시스템”입니다.

핵심 개념:

1. 디자인 시스템
버튼, 배지, 체크박스, 스위치, 테이블, 카드, 체크아웃 UI 같은 공통 UI 구성요소를 미리 정리해 둔 체계입니다.

2. 커스터마이징 가능한 테마
브랜드 색상, 다크모드, 간격, 스타일을 바꿔서 자기 서비스에 맞출 수 있다는 점을 강조합니다.

3. React 컴포넌트
페이지에 따르면 160개 이상의 접근성 지원 React 컴포넌트를 제공합니다.

4. 템플릿
바로 제품에 넣을 수 있는 페이지 템플릿을 제공한다고 설명합니다.

5. Agent-ready
CLI나 MCP를 통해 AI 에이전트가 프로젝트를 스캐폴딩하고, 템플릿을 찾고, 테마를 생성하고, 문서를 참고할 수 있게 만든다는 점이 중요합니다.

왜 중요한가:
일반 디자인 시스템은 사람이 문서를 보고 컴포넌트를 가져다 쓰는 방식입니다. Astryx는 여기에 “AI 에이전트가 이해하고 조작하기 쉬운 구조”를 붙인 쪽으로 보입니다. 즉, 앞으로 Codex/Cursor/Claude 같은 개발 에이전트가 UI를 만들 때 바로 활용할 수 있는 디자인 시스템을 목표로 한 것입니다.

페이지에서 볼 수 있는 예시:
• 테마 쇼케이스
• 상품 카드 UI
• 체크아웃 폼
• AI 채팅/주문 조회 UI
• 인벤토리 테이블
• 매출 카드
• 컴포넌트 탐색
• 템플릿/CLI/문서 링크

한 줄 정리:
Astryx는 “Meta 내부에서 성장한 디자인 시스템을, React 컴포넌트 + 테마 + 템플릿 + AI 에이전트 친화 문서/CLI 형태로 공개하려는 프로젝트”로 보면 됩니다.
