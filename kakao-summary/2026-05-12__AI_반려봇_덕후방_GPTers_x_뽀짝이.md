## 인사이트·노하우

- **Hermes 칸반 + Goal**: 패치노트에 칸반·Goal이 함께 있어 Goal을 묶어 쓰면 “헷갈리거나 막힐 때까지” 작업이 밀린다는 실사용 감이 공유됨. `/background`, `/goal` 활용이 인상적이었다는 후기도 있음.
- **오케스트레이션 역할**: “오케스트레이터 전용 프로필” 없이도 **시키면 그 에이전트가 오케스트레이터가 되기도 하고 워커도 될 수 있다**는 운영 팁. “욜로 백그라운드 goal로 칸반 작업 다 해달라” 하면 쭉 밀다 필요한 것만 물어본다는 식의 쓰임.
- **칸반 협업**: 칸반을 쓰면 봇끼리 역할·페르소나를 나눠 일 분배·전문 분야 위임이 쉽다는 설명.
- **OpenClaw와 Hermes 동일 PC**: OpenClaw 쓰는 환경에 Hermes를 같이 설치해 써도 된다는 답변. Hermes는 대략 **3~4GB** 정도 쓴다는 감.
- **Hermes GPT 프로바이더 오류(지연·초기화 고민)**: (1) Codex 등에 **저장된 인증키를 확인·정리하고 새로 인증**, (2) **`hermes setup`부터 다시**, (3) 모델 인증 시 **구독 Codex OAuth 선택을 물어보라**는 조언 흐름.
- **OpenAI 키 + Hermes “Codex” 프로바이더**: `platform.openai.com` 키를 쓰는 경우 UI에 “OpenAI”만 없고 **“OpenAI Codex”만 보였다**는 질문 → **Codex 선택 후 안내 URL에서 로그인·키 입력**으로 연결됨(구독으로 보임). “API 방식을 굳이 안 해도” 연결된 사례.
- **요금제 실무감**: **Go**는 사용량 리밋이 작아 **“거의 무료급”, 실사용은 어렵다**는 평 → **Plus/Pro 쪽이 많이 쓰인다**는 정리. 날씨·간단 크론 수준은 Codex로도 된다는 의견, 문서·작문은 Claude가 부드럽다는 취향 차이 언급.
- **Gemini + Hermes/OpenClaw**: **구독 연결은 막힐 수 있고(차단)**, API 쪽은 될 가능성 언급. “제미나이 OAuth로 Hermes 연결”은 구글 정책 때문에 어렵다는 느낌의 대화.
- **로컬 Ollama**: EEVE 등 쓰던 흐름에서 **Qwen 3.6 추천** 언급.
- **OpenClaw vs Hermes 난이도·카톡**: **Hermes가 조금 더 쉬운 느낌**, **카카오 연결 자체는 손에 익은 쪽 차이** 정도라는 의견.
- **PC 재시작 습관**: 윈도우는 며칠마다 껐다 켬, 맥 미니는 켜두는 경우, OpenClaw가 살아나지 않을 때, 개발 중 오류·새 로컬 웹 만들 때 등 **증상·상황 기준**으로 재부팅.
- **시크릿/토큰 입력**: 채팅으로 넘기는 것도 “가능하긴 한데” **권장되지 않는다**는 공감. **설정 파일(.env 등)에 직접**, **터미널 config의 토큰 입력 UI**, 또는 **AI에게 어디에 넣으면 되는지 물어 경로 파악** 등 사례.
- **AI 동영상 품질**: **1080으로 생성**하거나 **업스케일링** 언급.
- **카카오 에이전트 계정**: **2번(투넘버)·업무용 별도 계정**으로 구성한 사례—비공식/데이터 읽힘 리스크 인식.
- **노션 장애 확인**: [notion-status.com](https://www.notion-status.com/)로 상태 확인 후 “장애 맞다”는 확인 흐름. “자주는 아니다”는 완화 의견.
- **플레이스토어 환불(마누스 등)**: 고객센터·플레이스토어 경로로 **환불 + 4천 크레딧 잔존** 사례 언급(개인 경험, 정책은 변동 가능).
- **칸반 워커·스킬·자가진화(깊은 정리)**: `kanban-worker` 스킬이 워커 실행 시 자동 로드되고, 태스크별로 추가 스킬을 붙이는 구조가 있다는 식의 기술 설명이 길게 실림. 다만 **“배운 것 → 검토 → 스킬 승격”이 운영 규칙으로 완전 표준화된 상태는 아니다**는 결론과, **`reusable_lesson` 메타데이터 표준** 같은 개선 아이디어 제안.
- **멀티 오케스트레이터**: 4봇이 같은 역할을 **런타임만 다르게** 돌며, 예를 들어 **작업마다 다른 봇이 오케스트레이터·나머지는 워커**, OS/프로젝트별로 칸반이 분리되는 식의 운영 사례.
- **Tailscale**: 여러 PC의 에이전트를 묶이면 **그들로도 칸반** 가능하다는 언급. “테일스케일 하고 싶다”면 IP 복붙 안내를 받는다는 농담성 팁.
- **리서치가 끝나지 않을 때**: **Max turns 조정**, **칸반 작업을 더 잘게 쪼개기** 제안. Hermes가 범위를 넓혀 멈추는 경우 **“야단”(범위 축소 지시)**이 필요하다는 감.
- **Hermes Curator(스킬 정리) 기본값**: **7일 주기**, **30일 미사용 stale**, **90일 미사용 archive**, **번들/허브 설치 스킬은 건드리지 않고 에이전트가 만든 스킬만** 관리 대상이라는 설명.
- **Claude Code `/goal`**: 버전업으로 `/goal` 유사 기능이 생겼다는 언급—**완료될 때까지 멈추지 않고 진행**하는 명령 계열로 Codex 쪽과 대비.

## 자동화·시스템 적용 후보

- **칸반 + 백그라운드 Goal**: 한 번에 작업을 “보드에 올리고 goal로 밀기” 패턴은 반복 업무·에이전트 운영에 그대로 이식 가능.
- **주기적 “채팅/DB → AI 인사이트만 추출”**: 대화에 등장한 것처럼 **Hermes에게 읽히게 한 뒤 2시간/구간 단위로 요약**하는 크론형 워크플로(프롬프트·해상도·모델 먼저 맞추라는 운영 코멘트 포함).
- **칸반 worker 완료 메타데이터 표준화**: 긴 메시지에 나온 **`reusable_lesson` JSON 필드**처럼, 학습을 스킬로 승격하기 위한 **구조화된 완료 보고 템플릿**.
- **Curator·스킬 난립 방지**: “워커가 멋대로 스kil 만들면 품질 문제”라는 우려와 맞물려 **리뷰어·큐레이터 역할**을 운영 규칙으로 두는 패턴.

응 오빠, 확인해봤어. 답은 이렇게야:

결론

추가 설정이 “완전히 새로” 필요한 상태는 아니야.  
지금도 칸반 에이전트들은 어느 정도 자가진화할 수 있는 기반이 있어.

다만, 오빠가 말한 이상적인 형태:

worker가 반복 작업에서 배운 점을 남김
→ reviewer/희수가 검토
→ 좋은 절차는 skill로 승격
→ 다음 worker가 자동으로 더 잘함
이 루프는 부분적으로만 돌아가고 있고, 아직 “정책으로 강제된 완성형 루프”는 아니야.

---

지금 이미 되고 있는 것

1. 칸반 worker들은 자동으로 kanban-worker 스킬을 로드함

실제 코드 확인했어. dispatcher가 worker를 띄울 때 이렇게 실행해:

hermes -p <profile> --skills kanban-worker chat -q "work kanban task <id>"
즉 researcher, analyst, ops, reviewer, backend-eng, writer 같은 애들은 작업 시작 시 기본적으로:

- 칸반 작업 방식
- 완료 summary/metadata 작성법
- block/retry 방식
- worker pitfall

이런 걸 알고 시작해.

이건 이미 되어 있어.

---

2. task별 추가 스킬도 붙일 수 있음

칸반 task에는 skills 필드가 있고, dispatcher가 그걸 worker 실행 시 추가로 붙여줘.

예:

kanban-worker + hermes-agent
kanban-worker + openclaw-gateway-debugging
kanban-worker + lawwatch-legislative-mcp
이런 식으로 가능해.

즉 특정 작업에 맞는 전문 스킬을 worker에게 먹이는 구조는 이미 있음.

---

3. worker 프로필들은 memory/skills 설정도 있음

확인한 worker 프로필들:

researcher
analyst
reviewer
backend-eng
ops
writer
2b
대부분:

- memory enabled
- skills 설정 존재
- kanban dispatcher enabled

상태야.

특히 analyst, reviewer, backend-eng, ops, writer는 memory provider가 honcho로 되어 있어.

그래서 “작업 경험을 남기고 다음에 활용하는 기반”은 있어.

---

아직 완성형으로 안 되어 있는 것

근데 오빠가 상상한 “장인들이 계속 자기 매뉴얼을 발전시키는 구조”는 아직 자동으로 빡세게 강제되진 않아.

현재 worker들은 주로:

작업 수행
→ summary/metadata 남김
→ complete/block
까지가 중심이야.

하지만 매번 자동으로:

이 작업에서 배운 절차를 reusable_lesson으로 정리해라
스킬 후보를 제안해라
희수에게 skill promotion 요청해라
reviewer가 검토해라
희수가 skill_manage로 반영해라
이 흐름이 모든 task에 표준화되어 있진 않아.

즉 지금은:

가능한 기반은 있음
하지만 자가진화 루프가 운영규칙으로 완전히 자동화된 건 아님
이게 정확해.

---

그래서 추가 설정이 필요하냐?

단순히 “worker도 skill을 쓸 수 있냐?”

아니, 추가 설정 거의 필요 없음.  
이미 kanban-worker 자동 로드되고, task별 스킬 추가도 가능해.

“worker들이 배운 걸 체계적으로 skill로 승격시키게 할 거냐?”

응, 이건 추가 운영 규칙/작은 패치가 있으면 좋아.

필요한 건 대형 설정이라기보단 이런 것들이야:

1. worker 완료 metadata 표준 추가
2. reviewer가 reusable lesson 검토
3. 희수가 skill 후보를 모아 승격
4. skill 난립 방지 규칙

---

내가 추천하는 추가 규칙

worker 완료 시 metadata에 이 필드를 표준으로 넣게 하면 좋아.

{
  "reusable_lesson": {
    "has_lesson": true,
    "summary": "다음에도 반복될 수 있는 절차 요약",
    "candidate_skill": "openclaw-gateway-debugging",
    "should_create_or_patch_skill": true,
    "confidence": "high",
    "evidence": ["실행한 명령", "검증 결과", "실패한 우회로"]
  }
}
그러면 희수가 나중에 task 결과를 보면서:

아 이건 스킬로 남길 가치 있음
→ skill_manage로 생성/패치
할 수 있어.

---

내 추천 구조

나는 worker에게 바로 자유롭게 스킬을 쓰게 하는 것보단 이게 좋아.

worker: 작업하면서 배운 점 기록
reviewer: 그 절차가 진짜 맞는지 검증
희수: 검증된 것만 skill로 승격
왜냐면 worker가 직접 막 스킬을 만들면:

- 중복 스킬 생김
- 임시 해결법이 영구 규칙이 됨
- 검증 안 된 명령이 매뉴얼화됨
- 공유 스킬이 지저분해짐

이럴 수 있어.

그래서 자가진화는 하되, 희수가 큐레이터 역할을 하는 게 안전해.

---

현재 상태를 한 줄로 말하면

칸반 에이전트들은 이미 스킬을 읽고, task별 스킬을 받을 수 있고, memory 기반도 있다.
하지만 “배운 점 → 검토 → 스킬 승격” 루프는 아직 표준 운영규칙으로 완전히 고정되어 있지 않다.
오빠가 원하면 내가 다음 단계로 Kanban worker 자가진화 규칙을 패치할 수 있어.

패치 방향은 작게 잡으면 돼:

1. kanban-worker 스킬에 reusable_lesson metadata 규칙 추가
2. kanban-orchestrator 스킬에 “스킬 후보 수집/승격” 규칙 추가
3. reviewer task가 should_create_or_patch_skill=true를 검토하게 함
4. 희수만 최종 skill_manage 수행

이렇게 하면 HQ 애들이 진짜로 “일하면서 배우는 작은 장인들”처럼 자라기 시작해.
개발특화 이런애들은 뭔가 스킬을 잘 짜주면 좋을거 같기도 한데
잘 짜주고 싶어도 개발을 모르니까 맡겨야 겠네요 ㅋㅋ
