# AI 요약

## 인사이트·노하우

- **카카오톡 봇 구현 방식 비교**: 공식 API 부재로 두 가지 비공식 경로만 존재.
  - **메신저봇R** (Android 폰 기반): 앱 내 `bot.addListener(Event.MESSAGE)` 로 메시지 감지 → GCP VM 등 외부 서버와 연동. 안정적이나 Android 폰 상시 구동 필요.
  - **카카오클로(OpenClaw) 방식**: 안 쓰는 서브계정 + kakaocli/kmsg CLI 로 SQLite DB 폴링하여 메시지 수신 후 에이전트에 전달. kakaocli가 동작 안 하는 케이스 있음, kmsg는 동작하나 응답이 느림.
  - 두 방식 병행 운용도 가능 (메신저봇R 버전 + 오픈클로봇 버전 별도 유지).

- **카카오톡 계정 전략 (봇용)**: 본계정 사용 시 밴 위험 있음. 듀얼번호 또는 서브폰 별도 계정 권장. 해외 회선은 오픈채팅 기능 사용 불가 (한국 회선만 가능). 실운용자는 카카오 계정 3개 (메인/듀얼/서브)로 테스트 환경 분리.

- **카카오톡 봇 업데이트 후 중단 이슈**: 카카오톡 앱 업데이트로 일부 비공식 봇이 중단됨. `agents` 에서 message read는 되지만 message send가 안 되는 케이스 보고됨. 재로그인 후 "문제 찾아서 복구해달라"고 요청하면 복구 가능한 경우도 있음. 일부는 영향 없이 계속 동작 중 — 구성 방식에 따라 차이 있는 것으로 추정.

- **카카오클로 아키텍처 예시 (실운용)**:
  ```
  카카오톡 앱 (Mac)
    ↓ kakaocli sync --follow (SQLite DB 폴링, 1초 간격)
  bridge.py (트리거 감지 + 채팅방 필터)
    ↓ openclaw agent --agent <name>
  agent-kakaotalk message send (답장)
  ```
  단, `카카오에이전트의 message send`를 직접 호출하는 방식은 비권장으로 확인됨.

- **카카오톡 봇 응답 지연 대처법**: Codex 등 LLM 특성상 응답이 느릴 수 있음 → 쿨타임을 명시적으로 설정하고 사용자에게 미리 고지하면 자연스럽게 수용됨 ("아~ 그렇구나").

- **오픈채팅방 지식관리 활용**: 혼자 들어간 오픈채팅방에 봇만 두면 개인 지식관리 도구로 활용 가능. 대화 요약 후 Obsidian으로 자동 저장하는 워크플로 실운용 중. Mac의 경우 대화 내보내기 자동화도 가능.

- **BTC 자동매매 실운용 사례**: 가변형 하이브리드 모델 + DeepSeek API 조합으로 5/30~6/8 기간 +6.8% 수익. Buy & Hold 대비 -10% 대비 상대 수익률 약 +19%p. DeepSeek API는 비용이 거의 없는 수준.

- **AI 토큰 사용량 거버넌스 (기업 사례)**:
  - 아마존: 토큰 낭비 방지 위해 사용량 순위판 폐쇄.
  - 우버: 4개월 만에 연간 AI 예산 소진 → 직원 1인당 월 $1,500 한도 적용.
  - 교훈: "AI 많이 쓰기 = 생산성"이라는 전제에 균열. 사용량 자체가 성과가 아님.

- **하네스 실력이 개발자 핵심 경쟁력**: "앞으로 개발자의 진짜 경쟁력은 코딩 실력이 아니라 하네스 실력에 달려있다. 하네스를 짜본 경험과 실력의 차이가 개발자의 진짜 경쟁력의 차이가 될 것."

## 반응이 컸던 화제

- **카카오톡 봇 업데이트 후 중단 여부**: 여러 명이 "중단됐다", "안 된다", "잘 된다"로 엇갈린 경험을 공유하며 활발히 논의. 구성 방식(메신저봇R vs 오픈클로 방식)에 따라 결과가 다른 것으로 보여 정보 수요가 높았음.

- **카카오톡 봇 아키텍처 공유**: [P54]가 자신의 구성을 코드 블록으로 공유하자 여러 명이 질문 이어감. DB 폴링 방식의 응답 속도 문제, 대안 CLI(kakaocli vs kmsg) 등으로 토론 확장.

## 자동화·시스템 적용 후보

- **카카오톡 오픈채팅 + 봇 개인 지식관리 파이프라인**: 혼자 들어간 오픈채팅에 봇 배치 → 대화 요약 자동화 → Obsidian 저장. Mac 환경에서 대화 내보내기 자동화 연계 가능.

- **카카오클로 아키텍처 패턴**: `kakaocli/kmsg SQLite 폴링 → bridge.py 트리거 감지 + 채팅방 필터 → openclaw agent 호출 → 답장` 흐름. 트리거에 쿨타임 부여하여 LLM 응답 지연을 사용자 경험상 자연스럽게 처리.

- **AI 사용량 비용 한도 가드**: 기업 운용 사례(우버 월 $1,500/인)처럼, 개인/팀 시스템에서도 per-user 일일·월간 토큰 한도 설정 패턴 도입 고려.

## 질문·미해결

- kakaocli가 일부 환경에서 동작하지 않는 원인 불명. kmsg는 동작하나 응답이 느린 이슈 — 개선 방법 미확인.
- 카카오톡 업데이트 후 `message send`가 막히는 케이스와 정상 케이스의 구체적인 차이 조건 불명.
- 헤르메스가 "반려봇(인격 봇 다수 운용)"에 적합한지에 대한 명확한 답 미제공 ("둘 다 상관없다"는 정도로만 정리됨).

이번 영상의 핵심은 “AI 에이전트가 PC·보안·금융·자동차·기업 업무 전반으로 빠르게 들어오고 있지만, 실제 운영 단계에서는 책임·비용·물리 환경 인식 같은 새로운 문제가 동시에 드러나고 있다”는 점입니다.

화자가 강조한 흐름 위주로 정리하면 이렇습니다.

<1. 엔비디아가 AI PC 시장을 직접 노린다>

엔비디아가 PC 시장에 직접 뛰어들었습니다.

젠슨 황은 대만 컴퓨텍스에서 “40년 만에 PC를 다시 발명하고 있다”고 선언했습니다.

핵심은 GPU와 CPU를 하나로 합친 엔비디아의 새 칩, RTX 스파크입니다.

이 칩이 들어간 PC에서는 클라우드가 아니라 로컬 환경에서 AI를 돌릴 수 있습니다.

화자는 이것을 AI 에이전트 시대에 맞춰 엔비디아가 미래 AI PC 시장을 선점하려는 전략으로 봅니다.

지금까지 PC의 중심이 운영체제였다면, 앞으로는 AI 에이전트가 앱과 서비스를 넘나들며 일을 처리하는 구조로 바뀔 수 있다는 관점입니다.

<2. 마이크로소프트도 “AI 에이전트 기기”를 준비 중이다>

마이크로소프트도 AI 에이전트 시대에 맞춘 새로운 컨셉 기기를 공개했습니다.

책상 위에 두는 스마트 디스플레이, 목에 거는 형태의 기기 등이 언급됩니다.

화자의 포인트는 기기의 형태 자체보다, “PC 생태계의 중심이 운영체제에서 AI 에이전트로 이동할 수 있다”는 점입니다.

과거에는 Windows 같은 운영체제가 PC 사용 경험을 지배했지만, 앞으로는 사용자가 직접 앱을 열고 조작하기보다 AI 에이전트가 대신 앱과 서비스를 넘나들 수 있다는 뜻입니다.

<3. 한국도 앤트로픽 ‘글래스윙 프로젝트’에 참여한다>

한국 정부가 앤트로픽의 미토스 모델 접근권을 얻었습니다.

미토스는 사이버 공격 능력이 매우 강력해 일반 출시하지 않고, 일부 기관에만 제공되는 모델로 소개됩니다.

이 프로그램의 이름이 글래스윙 프로젝트입니다.

한국에서는 과기정통부를 통해 한국인터넷진흥원이 참여합니다.

민간 기업으로는 삼성전자, SK하이닉스, SK텔레콤 등이 포함된 것으로 전해졌습니다.

화자는 사이버 침해 사고가 잦은 한국에서 이 모델이 얼마나 많은 취약점을 찾아낼 수 있을지 주목합니다.

즉, AI가 공격 도구가 될 위험도 있지만, 동시에 선제적 방어 도구로 쓰일 수 있다는 점을 강조합니다.

<4. 금융도 AI 에이전트가 직접 거래하는 단계로 들어갔다>

주식 투자에도 AI 에이전트 시대가 열렸습니다.

미국 주식 거래 플랫폼 로빈후드는 ChatGPT, Claude 같은 외부 AI를 개인 증권 계좌에 연결해 자율 매매를 맡기는 서비스를 공개했습니다.

개인 투자자만의 변화가 아닙니다.

모건스탠리도 기업 고객의 AI 에이전트에게 자사 주식 보상 관리 플랫폼을 개방했습니다.

사람이 직접 로그인하지 않아도, 개인과 기업의 AI 에이전트가 주식 관련 업무를 처리할 수 있게 된 것입니다.

화자가 강조한 핵심 문제는 책임 소재입니다.

AI가 금융 거래를 직접 처리하면 편의성은 커집니다.

하지만 손실이 났을 때 책임이 누구에게 있는지는 아직 명확하지 않습니다.

그럼에도 서비스는 이미 시작됐다는 점이 중요합니다.

<5. 웨이모는 침수 도로 인식 문제로 한계를 드러냈다>

구글의 자율주행 서비스 웨이모 차량이 물이 고인 도로에 그대로 진입해 고립되는 사례들이 보고되고 있습니다.

웨이모는 이미 침수도로 감지 문제로 약 3,800대에 대해 자발적 리콜을 실시했습니다.

그럼에도 문제가 계속 발생하고 있습니다.

결국 웨이모는 로보택시의 고속도로 운행을 잠정 중단했습니다.

웨이모 차량은 레이더와 라이다 센서로 주변 환경을 파악하지만, 물 웅덩이의 깊이를 추정하는 데에는 한계가 있다는 점이 드러났습니다.

화자는 “과연 웨이모가 이 문제를 극복할 수 있을까?”라는 질문을 던집니다.

AI와 자율주행이 발전해도, 물·침수·깊이 같은 물리 세계의 복잡성을 인식하는 것은 여전히 어려운 과제라는 메시지입니다.

3,800대 | 웨이모 침수도로 감지 문제 관련 자발적 리콜 규모
잠정 중단 | 웨이모 로보택시 고속도로 운행

<6. 실리콘밸리의 ‘토큰 많이 쓰기’ 문화에 제동이 걸렸다>

몇 달 전까지만 해도 실리콘밸리 기업들은 AI를 최대한 많이 쓰라고 장려했습니다.

하지만 이제 분위기가 바뀌고 있습니다.

아마존은 직원들이 AI 토큰 사용량 순위를 올리기 위해 토큰을 낭비하자, 사용량을 보여주는 상황판을 폐쇄했습니다.

우버도 직원들에게 AI 사용량을 내부 리더보드로 경쟁시키며 사용을 장려했습니다.

그러나 임직원들이 단 4개월 만에 연간 AI 예산을 소진할 정도로 토큰을 많이 써버리자, 직원 1명당 월 1,500달러 한도를 걸었습니다.

화자의 핵심 메시지는 “AI를 많이 쓸수록 생산성이 오른다”는 실리콘밸리의 믿음에 균열이 생기고 있다는 점입니다.

AI 사용량 자체가 성과가 아니라는 문제의식입니다.

4개월 | 우버 임직원들이 연간 AI 예산을 소진한 기간
1,500달러 | 우버 직원 1인당 월간 AI 사용 한도

PC에서는 로컬 AI 에이전트가 돌아가고, 보안에서는 강력한 AI 모델이 취약점을 찾으며, 금융에서는 AI가 직접 계좌 업무를 처리하고, 자율주행에서는 현실 세계의 한계와 마주하고 있습니다.

동시에 기업 내부에서는 “AI를 많이 쓰는 것”이 곧 생산성이라는 믿음이 흔들리고 있습니다.

즉, AI 에이전트 시대는 이미 시작됐지만, 책임·비용·물리 세계 인식·보안 통제라는 새로운 병목이 함께 등장하고 있다는 것이 화자의 핵심 메시지입니다.
