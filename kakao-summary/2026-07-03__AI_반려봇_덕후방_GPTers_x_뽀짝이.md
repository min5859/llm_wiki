## 인사이트·노하우

### Claude Fable 시스템 프롬프트 누출본으로 "Fable 모드" 프롬프트 만들기

- [누출된 Fable 시스템 프롬프트](https://github.com/asgeirtj/system_prompts_leaks/blob/main/Anthropic/claude-fable-5.md)를 기반으로, GPT 등 다른 모델에게 "Fable처럼 동작하라"는 커스텀 프롬프트를 주입하는 시도가 여럿 이루어지고 있음.
- 한계: 모델 자체 성능 차이 때문에 완전 재현은 불가. "기아차에 벤츠 마크 붙이는 느낌". CoT 사고과정 패턴을 참고해 워크플로우를 강제할 수는 있으나 성능 갭은 그대로.
- 실용 수준: GPT 5.5에 적용 시 "프롬프트 먹여놓은 정도"의 개선은 체감. 일부 참가자는 로컬 LLM에도 적용 실험 예정.

### Fable 토큰 절약 전략: 계획은 Fable, 구현은 Codex

- Fable로 구현까지 맡기면 토큰이 1시간 만에 소진됨.
- 패턴: **Fable(설계) → Markdown 산출물 → Codex(구현)**
- 추가로 Codex 앞단에 토큰 절약 전처리 레이어를 두는 방식도 사용 중.
- 기존에는 Opus(설계) → Codex(구현)으로 하루종일 써도 여유가 있었으나 Fable은 훨씬 빠르게 소진됨.

### 에이전트 간 긴 내용 전달 시 "빈 내용" 오류 해결

- 증상: 클로드 코드·Hermes에서 복사한 긴 내용을 GPT/Claude 웹에 붙여넣을 때 "빈 내용"이라는 오류 발생.
- 해결: 내용을 `.md` 파일로 저장 후 경로를 입력하거나 파일로 전달.
- 워크플로우: 계획서를 MD로 작성 → 다른 에이전트에서 해당 MD 기반으로 개발 진행.

### Hermes 하트비트 크론의 토큰 과소비 문제

- 증상: 토큰이 예상보다 빠르게 소진 → 조사하니 하트비트 캐시가 주요 원인.
- 하트비트에 대시보드 체크 등 무거운 작업을 붙여두면 컨텍스트가 쌓여 토큰 증가.
- 대응 방법:
  1. 하트비트 작업을 별도 경량 워크플로우로 분리.
  2. 하트비트용 모델을 별도로 사용.
  3. 토큰이 일정량 이상 사용되면 자동 알림·점검 로직 추가 검토.

### Hermes: workspace path vs cwd path 구분

- **workspace path**: 작업 대상 프로젝트의 기준 경로 (프로젝트 선택·세션 그룹 등에 사용).
- **cwd path**: 명령어·파일 작업이 실제 실행되는 현재 위치.
- 혼동 시 발생하는 문제: 상대경로(`README.md` 등)가 workspace가 아닌 cwd 기준으로 해석되어 엉뚱한 폴더에 파일이 생기거나 다른 프로젝트 파일이 수정됨.
- cwd가 예상과 다른 주요 원인:
  - CLI에서 hermes를 실행한 위치
  - Gateway/Telegram/Discord 등 메시징 실행 환경
  - Docker·컨테이너 샌드박스 (로컬 `~/my-app` ≠ 컨테이너 내 `/workspace`)
  - Cron/백그라운드 잡 (workdir 미지정 시 임의 위치)
  - `terminal.cwd` 설정이 `.` 또는 오래된 경로로 남아 있는 경우
- **실전 습관: 파일 작업 전 반드시 workspace path와 cwd path를 분리 확인.**

### Hermes Google Workspace 연동 인증 풀림

- 증상: Hermes에서 GWS CLI로 Gmail·Drive·Sheets를 연동하면 인증이 자주 풀림.
- 원인 추정: Google의 "임시 앱(미검증 OAuth 앱)" 상태로 연결 시 세션이 자주 만료됨.
- 미해결 상태; 다른 연동 방법 모색 중.

## 반응이 컸던 화제

- **Fable 시스템 프롬프트 기반 GPT "Fable 모드" 제작** — 여러 참가자가 독립적으로 같은 아이디어를 시도하고 있었다는 사실이 드러나며 공감과 경험 공유로 이어짐. 모델 성능 한계에 대한 솔직한 평가(프롬프트로는 근본적 성능차를 메울 수 없다)도 함께 논의됨.

- **Fable vs Opus 토큰 소비 비교** — Fable이 Opus 대비 토큰을 훨씬 빠르게 소진한다는 구체적 경험(1시간컷 vs 하루종일 가능)이 공유되어 여러 참가자의 공감 및 대안 워크플로우 논의로 이어짐.

Hermes에서 workspace path와 cwd path는 다를 수 있습니다

Hermes를 쓰다 보면 워크스페이스 경로와 cwd 경로가 서로 다르게 잡혀 있는 경우가 있습니다.  
처음에는 둘 다 “작업 폴더”처럼 보여서 헷갈리지만, 실제 역할은 다릅니다.

한 문장으로 정리하면 이렇습니다.

 **워크스페이스는 “내가 어떤 프로젝트를 하고 있는지”를 나타내는 기준이고,  
 cwd는 “명령어와 파일 작업이 실제로 실행되는 현재 위치”입니다.**

---

쉬운 비유

집으로 비유하면:

- workspace path  
  → “오늘 작업할 집 주소”  
  → 예: my-app 프로젝트

- cwd path  
  → “지금 내가 실제로 서 있는 방”  
  → 예: /root, /tmp, /workspace, ~/Downloads, Hermes 설치 폴더 등

집 주소는 A인데, 실제로 서 있는 방이 B일 수 있습니다.  
이 상태에서 “여기 있는 파일 수정해줘”라고 하면 Hermes는 cwd 기준으로 파일을 찾을 수 있습니다.

---

차이점 표

의미
• workspace path: 작업하려는 프로젝트/세션의 기준 경로
• cwd path: 현재 명령어가 실행되는 실제 위치

주로 쓰이는 곳
• workspace path: 프로젝트 구분, 세션 그룹, UI/워크스페이스 표시
• cwd path: 터미널 명령, 파일 읽기/쓰기, 상대경로 해석

예시
• workspace path: /Users/me/project-a
• cwd path: /root, /workspace, /tmp, .

바뀌는 방식
• workspace path: 프로젝트 선택, 워크스페이스 전환
• cwd path: Hermes 실행 위치, terminal.cwd, workdir, cd 명령 등

위험
• workspace path: 프로젝트를 잘못 고를 수 있음
• cwd path: 파일을 엉뚱한 폴더에 만들거나 수정할 수 있음

---

왜 예상 밖 경로가 나오나요?

Hermes에서는 실행 환경에 따라 cwd가 다르게 잡힐 수 있습니다.

예를 들면:

1. CLI에서 Hermes를 실행한 위치
   - 터미널에서 hermes를 실행한 폴더가 기본 cwd가 될 수 있습니다.

2. Gateway / Telegram / Discord / Slack 실행 환경
   - 메시징으로 호출된 Hermes는 사용자가 보고 있는 프로젝트 폴더가 아니라, gateway 프로세스가 시작된 위치를 기준으로 잡을 수 있습니다.

3. Docker / Modal / Daytona 같은 샌드박스
   - 로컬 프로젝트는 ~/my-app인데, 컨테이너 안에서는 /workspace나 /root가 cwd일 수 있습니다.

4. Cron job / background job
   - 예약 작업은 사용자의 현재 터미널 위치와 무관하게 실행됩니다.
   - 그래서 별도 workdir를 지정하지 않으면 예상과 다른 위치에서 실행될 수 있습니다.

5. terminal.cwd 설정
   - Hermes 설정에는 터미널 작업 기준 경로가 따로 있을 수 있습니다.
   - 이 값이 . 또는 오래된 경로로 남아 있으면 의외의 위치로 보일 수 있습니다.

---

가장 중요한 실전 포인트

Hermes에게 파일 작업을 시키기 전에 아래 두 가지를 분리해서 확인하는 것이 좋습니다.

1. 지금 workspace path는 어디인가?
2. 지금 cwd path는 어디인가?


특히 이런 요청을 하기 전에는 꼭 확인하는 것이 좋습니다.

- README.md 수정해줘
- 이 프로젝트 실행해줘
- 테스트 돌려줘
- package.json 고쳐줘
- 현재 폴더에 파일 만들어줘

왜냐하면 README.md 같은 상대경로는  
workspace가 아니라 cwd 기준으로 해석될 수 있기 때문입니다.

---

학습자용 체크 프롬프트

Hermes에게 이렇게 물어보면 좋습니다.

현재 Hermes 세션에서 workspace path와 cwd path를 분리해서 확인해줘.

확인할 것:
1. 현재 명령어가 실행되는 cwd
2. 현재 git 프로젝트 루트
3. Hermes terminal.cwd 설정
4. 현재 workspace/project 경로가 따로 있는지
5. workspace와 cwd가 다르면, 파일 작업이 어디에 적용되는지 예시로 설명

주의:
- .env, API 키, 토큰, 비밀번호는 출력하지 마.
- 추측하지 말고 실제 경로를 확인해서 말해줘.

 작업 전에는 항상 workspace와 cwd를 따로 확인한다.

엉뚱한 폴더에 파일이 생기거나, 다른 프로젝트 파일을 수정하는 실수를 크게 줄일 수 있습니다.
