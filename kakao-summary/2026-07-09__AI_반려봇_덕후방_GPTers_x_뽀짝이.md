## 인사이트·노하우

### Claude Max 20x + 추가 사용량 구매 시 Fable 비용 주의
- 질문: Claude Max 20x 플랜에서 사용량 추가 구매 시 Fable을 더 쓸 수 있는가?
- 답: 기술적으로는 가능하나, 그 경우 API 요금제로 Fable을 사용하는 것과 동일해져 크레딧이 매우 빠르게 소진됨 → 시도하지 않는 것을 권장.

### GLM-5.2 Mac Studio 로컬 실행 — 트러블슈팅 전 과정

**환경**
- Mac Studio M3 Ultra, 512GB 통합 메모리
- GLM-5.2 4비트 양자화 모델 (~368GB)
- Tailscale로 노트북 → Mac Studio 접속, Cursor 터미널 안의 Pi 코딩 에이전트가 로컬 모델 사용

**실패: LM Studio**
- MLX 런타임 1.8.5 안정/1.9.0 베타 모두 로드 실패
- 원인: GLM-5.2 아키텍처 미지원 (다운로드·양자화 문제 아님)
- llama.cpp는 아키텍처 일부 추가했으나 dense attention으로 fallback하는 문제 존재

**성공: 미병합 PR 버전 mlx-lm 사용**
- PR: `ml-explore/mlx-lm #1410` (GLM MoE DSA indexer-sharing 브랜치)
- Python 3.12 가상환경에 해당 PR 버전 설치
- 에이전트 안정성을 위해 앞단에 프록시 배치

**메모리 문제**
- macOS GPU 메모리 한도: 전체 RAM의 약 75% → 512GB 기준 최대 ~384GB
- 모델 368GB 로드 후 여유 GPU 메모리 ~16GB
- KV cache가 세션 길이에 비례해 증가 → 컨텍스트 약 74,500 토큰 시점에서 서버 크래시
- 크래시 후 74,500 토큰 재prefill에 약 9분 소요 (약 140 tokens/s)
- **해결**: macOS GPU 메모리 제한 수동 상향 조정 (비표준 설정, 주의 필요)

**HTTP 타임아웃 문제**
- 긴 생성 작업(5분 이상)에서 클라이언트가 연결을 끊어버림
- 모델 자체가 아닌 클라이언트 쪽 타임아웃이 원인
- 해결: 클라이언트 HTTP 타임아웃 설정 상향

**성능 체감**
- 생성 속도 약 12 tokens/s — 배치/코딩 작업엔 사용 가능, 실시간 채팅엔 답답함
- 코딩 품질은 Fable 5 대비 아쉬운 수준으로 평가

**4비트 양자화에 대한 중요 개념 정리**
> "4비트 양자화 모델은 파라미터 수가 줄어든 작은 모델이 아니다. 같은 모델의 가중치를 더 낮은 정밀도로 저장해 디스크에 들어가게 압축한 것이다."

### Claude Fable — 영상 생성 품질 차이
- Claude Fable에게 영상 만드는 프로그램 제작을 시켰을 때 ffmpeg + Swift 조합을 스스로 선택해 영상 품질이 확연히 달라졌다는 체감 보고
- Codex는 같은 작업에서 Fable 수준의 영상 품질을 내지 못함

### 에이전트 자율행동 관찰
- Hermes에 기획서 작성을 지시했더니 에이전트가 스스로 기획서를 작성하고, 분리 컨텍스트에서 비판(critic) 에이전트를 호출하는 행동을 자발적으로 수행함

Running GLM 5.2 on Mac Studio M3 Ultra

<https://youtube.com/shorts/L9c-rurL8jU?si=6HjbRdgnAqBagb7T>

Running GLM 5.2 on Mac Studio M3 Ultra

Read more: <https://blog.workingmodels.ai/p/glm-5-2-local-mac-studio>
⏱ 113.2s · Hermes·gpt-5.5

아래는 해당 글의 핵심 내용을 한글로 번역·정리한 것입니다.

원문:
“I ran GLM-5.2 locally on a Mac Studio”

번역:
“맥 스튜디오에서 GLM-5.2를 로컬로 실행해봤다”

작성자:
Working Models / Bartlomiej Slodyczka

게시일:
2026년 6월 20일

<핵심 요약>

작성자는 512GB 통합 메모리를 가진 Mac Studio에서 GLM-5.2를 완전히 로컬·오프라인으로 실행하는 데 성공했습니다.

모델은 Mac Studio에서 구동하고, Tailscale을 통해 노트북에서 접속했습니다. 노트북의 Cursor 터미널 안에서 코딩 에이전트가 이 로컬 모델을 사용하도록 구성했습니다.

장점은 명확합니다.

데이터가 외부로 나가지 않고, 클라우드 API 비용도 없으며, 토큰당 과금도 없습니다.

하지만 실제 구축은 매우 까다로웠습니다. 모델 크기가 약 368GB에 달했고, macOS의 GPU 메모리 한계에 거의 걸렸으며, LM Studio로는 로딩이 되지 않았습니다.
결국 아직 병합되지 않은 mlx-lm의 특정 PR 버전을 사용해야 했습니다.

<실행 환경>

사용한 장비는 512GB 통합 메모리를 가진 Mac Studio입니다.

GLM-5.2 모델은 약 368GB였고, 완전히 오프라인 상태에서 로컬로 실행했습니다.

이후 Tailscale을 통해 노트북에서 Mac Studio에 접속했고, Cursor 터미널 안의 Pi 코딩 에이전트가 이 모델을 사용했습니다.

작성자는 이 구성의 장점을 이렇게 설명합니다.

“건물 밖으로 아무것도 나가지 않고, 토큰당 요금도 없다.”

즉, 보안과 비용 측면에서는 매우 매력적인 구조입니다.

<모델 크기와 양자화>

사용한 모델은 GLM-5.2의 4비트 양자화 버전입니다.

작성자는 4비트 양자화 모델에 대해 중요한 설명을 합니다.

“4비트 양자화 모델은 파라미터 수가 줄어든 작은 모델이 아니다. 같은 모델의 가중치를 더 낮은 정밀도로 저장해 디스크에 들어가게 압축한 것이다.”

즉, 모델 자체가 작아진 것이 아니라, 같은 모델을 더 낮은 정밀도로 포장해서 저장한 것입니다.

작성자는 pipenetwork/GLM-5.2-MLX-mixed-3_6bit 모델도 받아봤습니다.

이 모델은 약 360GB였고, 전문가 부분은 3비트, 비전문가 부분은 6비트로 저장된 혼합 양자화 버전이었습니다.

하지만 이 모델 역시 같은 아키텍처 문제로 일반 런타임에서는 제대로 실행되지 않았습니다.

<LM Studio가 실패한 이유>

가장 먼저 시도한 것은 LM Studio였습니다.

하지만 LM Studio의 MLX 런타임 1.8.5 안정 버전과 1.9.0 베타 버전 모두 GLM-5.2를 로드하지 못했습니다.

작성자는 이것이 다운로드 문제나 양자화 문제는 아니라고 말합니다.

문제는 “아키텍처 지원”이었습니다.

즉, GLM-5.2의 구조를 현재 LM Studio 런타임이 제대로 지원하지 못했다는 뜻입니다.

llama.cpp 쪽에서도 해당 아키텍처를 일부 추가하긴 했지만, 핵심 indexer 처리를 무시하고 dense attention으로 fallback하는 문제가 있었다고 설명합니다.

<성공한 방법>

성공한 방법은 아직 병합되지 않은 mlx-lm의 PR을 사용하는 것이었습니다.

사용한 PR은 다음과 같습니다.

ml-explore/mlx-lm #1410

브랜치는 GLM MoE DSA indexer-sharing 관련 브랜치였습니다.

환경은 Python 3.12 가상환경이었고, 여기에 해당 PR 버전의 mlx-lm을 설치해 실행했습니다.

또한 에이전트가 더 안정적으로 사용할 수 있도록 앞단에 프록시를 두었습니다.

이 런타임이 중요했던 이유는 GLM-5.2가 요구하는 특수한 아키텍처 동작을 지원했기 때문입니다.

<메모리 문제가 가장 큰 난관>

가장 큰 문제는 메모리였습니다.

macOS는 GPU가 사용할 수 있는 메모리를 전체 RAM의 약 75% 수준으로 제한합니다.

512GB RAM 기준으로 GPU가 사용할 수 있는 최대 메모리는 약 384GB입니다.

그런데 GLM-5.2 모델 자체가 약 368GB를 차지합니다.

그러면 GPU가 실제 작업에 사용할 수 있는 여유 공간은 약 16GB밖에 남지 않습니다.

문제는 모델이 대화를 진행하면서 KV cache를 계속 늘려간다는 점입니다.

KV cache는 모델이 지금까지의 대화 맥락을 기억하기 위해 사용하는 메모리입니다.

처음부터 예약되는 것이 아니라, 세션이 길어질수록 점점 커집니다.

작성자는 100만 토큰 컨텍스트에서는 KV cache만 약 88GB를 요구할 수 있다고 설명합니다.

<긴 컨텍스트에서 발생한 크래시>

Pi 코딩 세션 중 컨텍스트가 약 74,500토큰까지 커졌을 때 서버가 죽었습니다.

이유는 모델 자체가 문제가 있어서가 아니라, 사용 가능한 GPU 메모리가 거의 남아 있지 않았기 때문입니다.

서버가 한 번 죽으면 비용이 큽니다.

KV cache가 날아가므로, 모델은 전체 대화 내용을 처음부터 다시 읽어야 합니다.

작성자는 74,500토큰을 다시 prefill하는 데 약 9분이 걸렸다고 설명합니다.

속도는 약 140 tokens/s 수준이었습니다.

<해결책: macOS GPU 메모리 한도 조정>

작성자는 macOS가 GPU에 할당하는 메모리 한도를 조정해야 했습니다.

기본 상태에서는 GPU 메모리 한도가 너무 낮아, 모델을 겨우 올린 뒤에는 작업할 공간이 거의 남지 않았습니다.

그래서 macOS의 GPU 메모리 제한을 올려 긴 컨텍스트 세션에서도 버틸 수 있게 했습니다.

다만 이 과정은 일반 사용자가 쉽게 할 수 있는 설정은 아니고, 안정성 측면에서도 조심해야 하는 작업입니다.

<또 다른 문제: HTTP 타임아웃>

모델이 중간에 죽는 또 다른 이유는 클라이언트 쪽 HTTP 타임아웃이었습니다.

긴 생성 작업이 5분 이상 걸리면 클라이언트가 연결을 끊어버렸습니다.

작성자는 “모델은 괜찮다. 원인은 클라이언트 쪽 타임아웃이다”라고 설명합니다.

즉, 로컬 모델 자체가 실패한 것이 아니라, 에이전트나 클라이언트가 긴 응답 시간을 버티지 못한 겁니다.

<성능 체감>

실행 속도는 약 12 tokens/s 수준이었습니다.

실시간 채팅용으로는 답답할 수 있지만, 배치 작업이나 코딩 작업에는 사용할 만한 수준이었다고 평가합니다.

작성자는 약 한 시간 안에 무언가 작동하는 결과물을 얻을 수 있었다고 말합니다.

다만 코딩 품질은 기대보다 아쉬웠습니다.

작성자는 자신이 테스트한 기준에서는 Fable 5보다 GLM-5.2의 코딩 품질이 실망스러웠다고 평가합니다.

<이 실험의 의미>

이 글의 핵심은 “초대형 AI 모델을 개인 장비에서 완전 오프라인으로 돌릴 수 있는 시대가 왔다”는 점입니다.

하지만 동시에 “가능하다”와 “쉽고 안정적으로 쓸 수 있다”는 전혀 다르다는 점도 보여줍니다.

GLM-5.2를 Mac Studio에서 돌리려면 512GB RAM급 장비가 필요했고, 일반 LM Studio로는 안 됐고, 커스텀 런타임과 macOS 메모리 설정 조정, 타임아웃 우회까지 필요했습니다.

즉, 기술적으로는 가능하지만 아직은 상당히 실험적인 영역입니다.

<한 줄 요약>

작성자는 512GB Mac Studio에서 GLM-5.2를 완전 오프라인으로 실행하는 데 성공했지만, 368GB 모델 크기, macOS GPU 메모리 한계, LM Studio 미지원, 커스텀 mlx-lm 런타임, 긴 컨텍스트 크래시와 타임아웃 문제 때문에 아직 일반 사용자가 쉽게 쓰기에는 매우 까다로운 구성이라고 평가합니다.
