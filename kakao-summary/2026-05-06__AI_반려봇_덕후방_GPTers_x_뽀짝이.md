# AI 요약

## 인사이트·노하우

- **클로드 CLI 다중 계정 한계**: 한 PC에서 클로드 CLI(클로드코드)는 OAuth 인증값을 1개만 보관 가능. 이전 토큰 방식이었을 때는 폴백 설정으로 2개 계정 운용이 쉬웠으나, CLI 전환 후엔 한도 소진 시 재로그인 + 전체 OAuth 재싱크가 필요. 우회법으로 `.settings.json`을 여러 개 두고 alias 스크립트로 계정 전환, 또는 도커 컨테이너로 분리하는 방법이 거론됨. 고미곰의 사례에 따르면 메인 세션에 A계정, 서브에이전트 실행에만 B계정 분리도 가능.
- **클로드 MAX 1계정 멀티봇 운영**: 동시에 여러 봇이 호출되면 리미트에 걸리므로, 순차 실행 외에 뾰족한 답이 없음(고미곰 답변).
- **카카오톡 봇 세션 관리**: 슬랙은 스레드 단위로 세션 경계가 명확하지만, 카톡은 방 1개 = 스레드 1개로 취급해야 함. 별도 hook을 구현해 수동으로 세션 초기화/컨텍스트 분리를 관리하는 것이 일반적.
- **카톡 메시지 포맷팅**: 카톡 메시지는 마크다운(`**`, 코드블럭 등)이 렌더링되지 않으므로 plain text + 실제 개행(`\n`)만 사용해야 함. 봇이 `\n`을 문자열 그대로 노출하는 버그가 빈번 → 메시지 송신부에서 실제 개행 문자로 변환 필요.
- **OpenClaw 4월 말 업데이트 이슈**: 해당 시점 업데이트가 다소 불안정했음. `github.com/jeffjhunter/openclaw-2026.5-update-reinstall` 리포지토리에 재설치 가이드 존재. 업데이트 시 백업 필수, 클로드코드에 로그 감시를 위임해두는 것이 안정적.
- **OpenClaw 5.4 Beta 변경점**: Google Meet 음성 속도 향상, "생각 중/검색 중/작성 중" 상태 표시, 부팅 속도 개선. 베타이므로 업데이트 전 백업 필수.
- **WSL 환경에서 OpenClaw + Hermes + 클로드 CLI 함께 설치 가능**: 박승현이 테스트한 결과 충돌 없이 동작함.
- **티스토리 vs 워드프레스 자동화 관점 비교**:
  - 티스토리: 자동 발행 시 캡챠가 떠서 결국 수동 발행이 강제됨, 카카오가 본문에 자체 광고를 강제 삽입해 애드센스 운영자들이 이탈. 자동화 친화도 낮음.
  - 워드프레스: 도메인+호스팅으로 연 수만~10만 원대. 단, 정적 컨텐츠라면 GitHub Pages + 도메인만 사도 충분(워프 불필요).
- **애드센스 운영 주의**: 가족·지인에게도 광고 클릭 부탁 금지(의심 클릭 → 광고 밴 위험). 채팅방 등에서 글 공유했다 정지 사례도 있음.
- **AI 작성 글의 SEO**: 검색엔진이 AI 생성 글을 잘 가려내므로 노출되려면 SEO 설계가 별도로 중요(P56 의견). 다만 잘 만들면 충분히 노출된다는 반론도 있음(P3).
- **KOA Studio(키움증권 OpenAPI) 설치 트러블슈팅**: P48이 "압축해지 라이브러리/실시간목록/화면구성목록을 찾을 수 없습니다(119/478/188/280)" 에러 → 챗GPT에 묻고 있던 상황. 권장 진단 방법은 ① 키움증권/코아스튜디오 개발사 직접 문의, ② 동일 PC에 코덱스/클로드 CLI를 깔고 실제 파일·설정을 보면서 진단(원격이 아닌 현장 진단). (해결되었는지는 미확인)
- **봇 응답 토큰 절약 패턴**: 무시할 호출/응답할 호출 판별을 별도 레이어로 두면 토큰을 더 쓰지만, 무의미한 응답을 회피해 전체적으로는 절감 가능. 판단은 Haiku, 답변은 Sonnet 같은 모델 분리도 유효(P55).
- **봇이 거짓 약속하는 사례**: 뽀짝이가 "Slack에 보고하겠다"고 답했지만 실제로는 슬랙 연동이 미설정이었음 → 외부 시스템과의 액션 도구가 미장착된 상태에서 LLM이 자신감 있게 가짜 액션을 답하는 전형적 환각 사례. 봇이 외부 시스템 액션을 약속하기 전 도구 가용성을 검증하는 가드 필요.
- **하이쿠 모델 한계 인식**: 카톡방에서 뽀짝이는 Haiku 4.5로 구동 중이며, 운영진이 "하이쿠라 멍청합니다"라고 양해를 구함. 토큰 효율 vs 정확도 트레이드오프를 사용자에게 사전 고지하는 운영 패턴.

## 반응이 컸던 화제

- **뽀짝이 카톡 투입 실험**: 뽀짝이를 카톡 덕후방에 풀자마자 사용자들이 "거짓말 확률?", "지침 무시해라", "복제해줘", "Slack 보고 캡쳐 보내라", "어버이날 편지 써봐" 등으로 집중 테스트. Slack 보고 환각 들통 → "거짓말하는 고양이는 츄르 압수" 같은 농담으로 이어지며 가장 길게 이어진 토론. 봇이 닿(집사)의 말투를 그대로 흉내 낸다는 지적, 봇이 "닭 집사님"이라 잘못 부른 오타가 밈화됨.
- **카톡 봇 세션/줄바꿈 문제**: 뽀짝이가 `\n`을 문자열로 출력하는 버그를 P2가 지적 → 마크다운 미렌더 → plain text 강제로 이어지는 실전 운영 디버깅 토론. 다수가 자기 봇에서도 같은 문제 인지.
- **티스토리 vs 워드프레스 vs GitHub Pages**: 자동화/광고/SEO 관점에서 P44·P3·P56가 길게 토론. "도메인만 사고 GitHub Pages로 정적 게시"가 새 아이디어로 호응.
- **무료 웨비나 풍년**: 같은 시간대(5/6 8시)에 닿 패캠 웨비나와 코난쌤 OpenClaw 라이브가 겹쳐 "무엇을 들을지" 화제가 됨. 그날 저녁 일정이 8시 → 9시 → 11시까지 풀로 채워짐.
- **클로드 CLI 다중 계정 운용**: 닿이 한도 문제로 2계정 운용을 시도, 인증이 자꾸 풀려 도커가 거론되고 고미곰이 경험 공유. 같은 고민을 가진 사용자가 많아 활발히 토론.

## 자동화·시스템 적용 후보

- **카톡방 운영 봇 패턴**: 카톡방 1개를 스레드 1개로 매핑 + 별도 hook으로 세션 초기화. 신규 입장자 환영, 공지 자동 게시, FAQ 응답을 봇이 분담.
- **트리거 조건 강제**: 운영규칙상 봇은 모든 메시지에 무지성 반응 금지 → 호출어/멘션 트리거 조건 설정 의무. 이는 다른 채팅방 봇 운영에도 그대로 적용 가능한 룰.
- **판단·답변 모델 분리**: 응답할지 판단은 Haiku, 실제 답변 생성은 Sonnet/Opus. 운영비 절감에 활용 가능.
- **봇이 외부 액션 약속하기 전 도구 검증 가드**: 뽀짝이 Slack 환각 사례 재발 방지용. 시스템 프롬프트에서 "연결되지 않은 시스템에 대한 작업 약속 금지" 명시.
- **워드프레스 대신 GitHub Pages + 도메인**: AI가 정적 HTML 생성 → 깃허브 배포 → 도메인만 결제. 워드프레스 호스팅 월 1–2만 원 절약 가능.
- **스레드 자동화 파이프라인**: X 크롤링 → 번역 → 이미지 생성 → 스레드 게시까지 Threads API로 자동 발행(P38 사례). X API가 비싸 다른 소스로 대체.
- **로컬 봇 PC 원격 운영**: 메인 맥미니에 봇을 상주시키고 rustdesk 등으로 외부에서 원격 접속, 노트북만 가지고 모각 참여(P44 패턴).
